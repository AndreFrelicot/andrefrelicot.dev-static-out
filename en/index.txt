Paper Shaders is a collection of WebGL fragment shaders designed to produce visual backgrounds, textures, halftone patterns, gradients, and generative effects through a small web API.

The original project, [paper-design/shaders](https://github.com/paper-design/shaders), is published under the **Apache 2.0** license.

The original project is web-first: GLSL, React, canvas, textures, images, and presets. My goal was different: keep the same visual catalog, but make it usable in native SwiftUI and Flutter apps through publishable packages and demo apps that expose every parameter.

The result is split into two ports:

- **PaperShaders** for Swift/Metal, with a MetalKit core and a SwiftUI layer.
- **paper_shaders** for Flutter, using Flutter runtime shaders and a typed Dart API.

This is not only a syntax translation exercise. A shader that compiles is not necessarily a shader that renders the same image. The real work was building a shared contract across three runtimes: WebGL, Metal, and Flutter.

## The Shared Contract

The most important part of the port is not the strangest shader function. It is the contract around the shader.

Every effect must receive the same families of parameters:

- a stable coordinate space;
- sizing modes such as `cover`, `contain`, and `none`;
- scale, rotation, origin, and offsets;
- a deterministic `frame`;
- speed;
- colors, images, or noise textures depending on the shader;
- named presets.

That contract makes the ports comparable. If `simplex-noise` or `spiral` do not receive the same UV space, origin, and notion of time, the discussion about shader fidelity starts too late: the systems are already different.

On the web, part of that logic naturally lives in the canvas, the vertex shader, and the WebGL environment. In Swift/Metal and Flutter, that responsibility has to be placed deliberately. The chosen approach is explicit: sizing and uniforms are part of the public API, and presets expose the same mental model across platforms.

```swift
SimplexNoiseView(
  params: .init(
    colors: ["#FF7B00", "#F9FFEB", "#330D82"],
    stepsPerColor: 2,
    softness: 0.12,
    sizing: ShaderSizingParams(fit: .cover, scale: 1)
  )
)
```

```dart
const SimplexNoiseView(
  params: SimplexNoiseParams(
    colors: ['#FF7B00', '#F9FFEB', '#330D82'],
    stepsPerColor: 2,
    softness: 0.12,
    sizing: ShaderSizing(fit: ShaderFit.cover, scale: 1),
  ),
)
```

Colors are plain hex strings on every platform, exactly as in the upstream React API: the same three strings produce the same render in React, SwiftUI, and Flutter, with no per-platform color conversion to get wrong.

Initialization code matters. A visual preset is not only a JSON blob: in a real app, you want to copy a Swift or Dart snippet, paste it into the project, and get the same render.

## Three Languages, Not Three Looks

The original WebGL implementation uses GLSL. The Swift port uses Metal Shading Language. Flutter also accepts GLSL as input, but within the constraints of Flutter runtime shaders compiled by the Flutter/Impeller toolchain.

On paper, the languages look close enough to make the task feel simple. In practice, details matter:

| Topic | WebGL / GLSL | Metal / MSL | Flutter runtime shader |
|---|---|---|---|
| Fragment coordinates | `gl_FragCoord` | coordinates passed to the fragment | `FlutterFragCoord()` |
| Floating modulo | `mod` | `fmod` + helpers | `mod` GLSL |
| Angles | `atan(y, x)` | `atan2(y, x)` | `atan(y, x)` |
| Bool uniforms | accepted | Metal types | often encoded as floats |
| Uniform arrays | `vec4 colors[10]` | typed buffers | strict Dart offsets |
| Textures | WebGL samplers | Metal textures | Flutter samplers, without the full WebGL ecosystem |

The hard part is not finding an equivalent function. It is preserving implicit conventions: orientation, pixel ratio, interpolation, precision, uniform order, samplers, and edge behavior.

The Flutter port has one advantage: a large portion of the original GLSL remains reusable. But it also has a strict constraint: every uniform has an exact position in the buffer expected by Flutter. A wrong offset does not always break compilation; it can turn a color into a scale, a boolean into a threshold, or an offset into a speed.

The Metal port is more explicit. You get types and native integration, but you pay for a shader-by-shader translation from GLSL to MSL.

## The Catalog in Six Families

Rather than a shader-by-shader walkthrough, here is the whole catalog grouped by visual family. Every image below is a golden render from the parity suite — each shader's default preset, exactly as the tests see it.

**Gradients** — `mesh-gradient`, `static-mesh-gradient`, `static-radial-gradient`, `grain-gradient`: soft color fields, from an animated color mesh to a radial ramp, plus a version roughened by film-like grain.

Mathematically, a mesh gradient is not a gradient but Shepard interpolation: every pixel averages the color points weighted by inverse distance, `w = 1/(d^3.5 + eps)`, while the points drift along `sin`/`cos` paths. Nothing is drawn — the softness is the average itself. `static-radial-gradient` is the most analytic of the family: it solves a ray–circle intersection (one quadratic per pixel) to produce an off-center radial falloff, and `grain-gradient` quantizes a shape field into a color ramp before roughening it with simplex grain.

**Organic noise** — `simplex-noise`, `perlin-noise`, `neuro-noise`, `smoke-ring`, `gem-smoke`, `warp`: continuous noise put to work — banded contours, speckled fields, glowing filaments, smoke, and marbled distortion.

Three kinds of "noise" coexist here. True simplex (the classic Ashima implementation, two octaves scrolling in opposite directions), true 3D Perlin (corner gradients, quintic interpolation `t^3(t(6t-15)+10)`) stacked as fbm — *fractional Brownian motion*, the same noise summed again at increasing frequency and decreasing amplitude, up to 8 octaves here — and fake: `neuro-noise` is not noise at all but a 15-iteration loop that rotates the frame by one radian per step and accumulates `sin`/`cos`; the interference pattern makes the filaments. The other big idea in the family is domain warping: `warp` distorts the coordinates themselves (`uv.x += s/i * cos(t + 1.5*i*uv.y)`, up to 20 iterations) before drawing plain stripes, and `smoke-ring` computes its fbm in polar coordinates to ripple the radius of a ring. Noise used as a value gives clouds; noise used as a displacement gives marble.

**Polar effects** — `spiral`, `swirl`, `god-rays`: pure `(angle, radius)` territory — concentric spirals, twisted color bands, and rays fanning out from a point.

The whole family fits in one formula: `fract(a*angle + b*f(radius))` after the polar transform. `spiral` takes `f(r) = pow(r, density)` — density tightens the arms; `swirl` takes a negative power, `pow(r, -twist)`, which twists N angular bands into a pinwheel; `god-rays` replaces the radial term with value noise sampled along the angle — noise in the angle is rays. The center is the sensitive spot: as the radius approaches zero the angle becomes unstable, and the `atan` discontinuity at ±π has to be hidden (`god-rays` blends two noise variants across the seam).

**Patterns and geometry** — `dot-grid`, `dot-orbit`, `voronoi`, `metaballs`, `color-panels`, `waves`, `pulsing-border`: repetition and cells — dot lattices, orbiting particles, Voronoi mosaics, merging metaballs, stacked translucent panels, zigzag waves, and a glowing border.

Here space is partitioned before anything is drawn: `floor`/`fract` cut the plane into cells (`dot-grid`), a 3×3 Voronoi loop finds the nearest cell center (`dot-orbit`), and `voronoi` runs a second pass with Inigo Quilez's border metric — the distance to the bisector between neighboring cells — to get clean edges. Each cell receives a hash that varies size, opacity, or color: repetition without uniformity. `metaballs` is a scalar field — a sum of `pow(1 - d/2, p)` falloffs, one per ball — thresholded at 0.4: the blobs merge because the fields add up before the threshold. And `color-panels` hides a real perspective projection, `z = y/(sin(a) - y*cos(a))`, inside a 2D fragment shader.

**Halftone and dithering** — `halftone-dots`, `halftone-cmyk`, `dithering`, `image-dithering`, `heatmap`: the quantization family — a continuous input, procedural or a user image, forced into dots, CMYK separations, dither patterns, or heat levels.

This is quantization against a threshold that is structured in space. `halftone-dots` turns luminance (Rec. 709 weights: `0.2126*R + 0.7152*G + 0.0722*B`) into a dot radius on a grid; `halftone-cmyk` separates the image into inks (`K = 1 - max(R, G, B)`) and screens each channel at its own angle — 15°, 75°, 0°, 45°, the actual print screen angles, which is where the rosettes come from. The two dithering shaders compare the signal against hardcoded 2×2, 4×4, and 8×8 Bayer matrices — `step(0.5, value + threshold - 0.5)`, one bit per pixel. `heatmap` closes the family: a heat scalar runs through a multi-stop color ramp — the lookup table (LUT) of thermal cameras. In every case the eye does the integration: dot size or dot density reads as tone again.

**Materials** — `paper-texture`, `fluted-glass`, `liquid-metal`, `water`: simulated surfaces, usually applied over a user image — paper, ribbed glass, liquid chrome, and water.

No real optics here — only UV displacement and simulated lighting. `fluted-glass` refracts by laterally shifting where the image is sampled, following a periodic per-flute profile (cubic, quadratic, or sine); `water` layers a low-frequency simplex (the waves) with a 6-iteration caustic accumulator, both displacing the UV, and the caustic squared becomes the highlight; `paper-texture` stacks noises (fibers as the gradient magnitude of an fbm, folds as distances to random points, drops as Worley cells), derives a normal by finite differences, and lights it with a plain Lambertian dot product; `liquid-metal` builds its chrome from a `fract()` stripe ramp whose red and blue channels are phase-shifted — chromatic dispersion as a subtraction. The question each pixel answers is never "where does the light go", but "which pixel of the image do I read, and how bright".

## The Small Math Behind Portability

Rather than going back through the shaders one by one, the more useful angle is the small set of mathematical families that power all six groups above, and explain why the same effect can remain portable.

### 1. UV Space

Most shaders start by turning the current pixel into normalized coordinates.

```glsl
vec2 uv = fragCoord / resolution;
vec2 p = uv - 0.5;
```

From there, the shader can move the origin, apply scale, rotate the basis, then feed those coordinates into a visual function. That is the foundation.

Rotation, for example, is not mysterious shader magic. It is a 2D matrix.

```glsl
mat2 rotate(float a) {
  float c = cos(a);
  float s = sin(a);
  return mat2(c, -s, s, c);
}
```

If this transform is not identical across platforms, the shader can be perfectly translated and still produce an offset, flipped, or stretched image.

### 2. Polar Coordinates

Effects such as `spiral`, `swirl`, radial gradients, and several distortions rely on converting `(x, y)` into `(angle, radius)`.

```glsl
float angle = atan(p.y, p.x);
float radius = length(p);
float bands = sin(angle + radius * frequency);
```

This transform is compact, but sensitive. Near the center, the radius approaches zero; the angle becomes unstable; small precision or anti-aliasing differences become visible. This is exactly the kind of shader where a port can look correct at first, then reveal a cross pattern, jagged edge, or strange oscillation on a specific backend.

### 3. Noise

Paper textures, grain, smoke, and organic gradients use noise as a controlled source of variation.

Noise is not only there to make the image dirty. It breaks numerical perfection. A surface that is too regular feels artificial; a bit of noise makes it closer to paper, ink, or material.

In this port, noise is also a practical constraint: some shaders use an embedded noise texture. The same PNG must be loaded with the same sampling conventions, otherwise the render drifts. A repeated 128×128 texture, filtered or sampled differently, can be enough to change the character of a shader.

### 4. Quantization

The `halftone`, `dithering`, and heatmap families do the opposite of continuous noise: they force the render into discrete levels.

```glsl
float stepped = floor(value * steps) / steps;
```

This operation is intentionally harsh. An image becomes a pattern, a color becomes a level, and a gradient becomes a set of bands. It is also the kind of effect that immediately exposes rounding differences between platforms.

### 5. GPU Derivatives

The most subtle part of the port was probably derivative-based anti-aliasing.

In GLSL, `fwidth(x)` estimates how much `x` changes across neighboring pixels. It is very useful for smoothing an edge:

```glsl
float edge = smoothstep(threshold - fwidth(v), threshold + fwidth(v), v);
```

But `fwidth` is tied to GPU execution. It depends on how the backend groups fragments and computes derivatives. For interactive rendering, that is usually what you want. For golden image parity, it is more complicated: two platforms can be visually very close while producing slightly different pixels along edges.

The compromise for tests was to replace some uses with a controlled deterministic derivative when parity required it. Flutter made part of that decision for me: its test runner rejects `fwidth(float)` outright, along with dynamic uniform-array indexing. This is not an artistic rewrite of the shader. It is a way to separate the behavior being tested from the local GPU implementation detail.

## Testing Shaders With Golden Images

The deterministic `frame` changes everything. If a shader depends only on uniforms, textures, and a frame number, it can be rendered offscreen and compared against a reference PNG.

The workflow looks like this:

```text
GLSL source -> manifest -> Swift/Metal port
                    \-> Flutter port

web golden PNG -> native renderer -> perceptual comparison
```

Each preset becomes a visual test case — 120 golden renders across the catalog. This is not perfect: image comparison does not prove that a shader is mathematically identical. But it catches the mistakes that matter for this kind of package:

- wrong uniform order;
- inverted color;
- incorrect rotation or origin;
- missing texture;
- sampling error;
- anti-aliasing drift;
- non-deterministic frame or speed.

For a shader library, the image is the contract.

## The Performance Cost of Precise Math

The first fullscreen runs of the Metal port on real devices were humbling: shaders that animate smoothly in a browser tab were dropping frames as native views. The GPU was not the problem. My parity setup was.

To keep golden renders aligned with WebGL, the Metal pipeline compiles with safe math and precise floating-point functions. Fast math lets the compiler contract and reassociate float operations and substitute approximate `sin`/`cos` — enough to decorrelate hash-based noise from the reference; `perlin-noise` octaves visibly drift. So precise mode is exactly what parity tests need. It is also significantly slower on shaders like `neuro-noise` that accumulate a lot of trigonometry per pixel — and a fullscreen Retina view pushes far more pixels than a typical browser canvas.

The package now treats parity and interactivity as two different rendering modes:

```swift
// Golden parity: precise float functions, fixed resolution.
ShaderRenderOptions.fixed

// Interactive: fast math, plus an internal resolution that
// adapts until the view sustains the target frame rate.
ShaderRenderOptions.adaptive(targetFrameRate: 60)
```

Fast math is the biggest lever; pixel count is the other one. The pixel budget mirrors upstream: a minimum pixel ratio and a cap on total rendered pixels. On top of that, adaptive mode runs a small feedback loop: frame durations are smoothed with an exponential moving average; when the smoothed value exceeds the target by more than about 12%, the internal resolution steps down; once there is comfortable headroom, it slowly steps back up, never below a configurable floor. Fragment-bound shaders scale almost linearly with pixel count, so a small resolution drop — usually invisible on an animated background — buys the frame rate back.

Each view can also report live render metrics (frames per second, drawable size, effective pixel ratio, adaptive scale, math mode), which the demo app displays as an overlay. And to keep the comparison honest, the workbench grew a standalone fullscreen web page that runs the upstream WebGL shaders on the same devices and reports FPS, canvas size, and device pixel ratio. Same shader, same preset, same screen — the only way to know whether the native port is actually slower, or simply rendering more pixels.

## Takeaway

Porting a shader is not only a syntax translation. It is a conservation problem: preserve a coordinate space, a notion of time, a texture, a rounding behavior, a threshold, and a visual intent.

The differences between GLSL, MSL, and Flutter are manageable. They mostly become visible when you demand parity, not only a good-looking approximation.

That is where the project becomes interesting: a small graphics library becomes a practical lab for render portability. The math stays modest, but it is everywhere: coordinates, rotations, noise, quantization, and derivatives. And precisely because the math is simple, it has to be aligned carefully.

## A heat wave, and the math of heat

Late June 2026, France was under a [brutal heat wave](https://meteofrance.com/actualites-et-dossiers/actualites/canicule-une-vague-de-chaleur-sinstalle-cette-semaine). The kind where the air shimmers above the asphalt, where everything slows down, and where you start staring at hot air rising off a rooftop wondering what shape it actually traces.

That shimmer is **convection**: a fluid heated from below, rising, cooling, sinking, rolling over itself. And convection has a famous set of equations attached to it — the ones Edward Lorenz wrote down in 1963 while trying to model exactly that, the atmosphere churning under heat.

So I built a toy out of it. **Lorenz Clash** is a generative audiovisual WebGPU experience for the mobile web — and one you become *part of*. It looks back at you through the front camera — the selfie lens on a phone, the webcam on a laptop — and that live video becomes a material that is sliced, stretched and dissolved along the 3D trajectory of the Lorenz attractor — the very curve born from the mathematics of heat. An app born from a heat wave, built on the equations of heat. That symmetry was too good not to chase.

This article is the deep dive: the **math** behind the chaos, the **WebGPU render pipeline** and its shader language, **WGSL** (WebGPU Shading Language), the **music-structure analysis** that lets it cut and breathe with a track, the **serverless video export** that records it all on-device, the **fight to survive a memory-tight tablet**, and the **Progressive Web App** (PWA) plumbing that makes it install fullscreen on iOS and Android. ## An experience you are part of

This is not an app you *operate* like a tool — it is one you step inside. The whole thing is built around a single relationship:

> The camera is the **matter**. The Lorenz attractor is the **force**. Your hand on the glass is the **gesture**. The audio is the **texture**.

It is deliberately **not** an audio visualizer, and **not** a camera filter. The camera (or, when it is off, a procedural source) is the *primary* visual material — it gets carried along the chaotic curve, aged, pixelated and dissolved.

And there is a quieter idea underneath. Because the camera always faces *you*, *you* are the matter — your face, your room, fed in as one more initial condition of a chaotic system. It is a small lesson in two halves. On one side, **responsibility**: what the piece becomes is made out of what you choose to put in front of it. On the other, the reminder that **it exceeds you** — this is the butterfly effect, so the same gesture never lands twice, and you can never quite predict, let alone steer, where the chaos carries your reflection. You are unmistakably *part of* the system and just as unmistakably not in control of it. If you have lived through a heat wave wondering how we got here, that double bind will feel familiar.

A few non-negotiable rules shaped every decision:

- **Everything runs on-device.** The video feed never leaves your phone — no upload, no backend, no tracking.
- **Camera and audio are optional, and motion is never required.** Deny the camera and the piece still works (a procedural source takes over). Mute the music and the visuals keep moving. Ignore the touch gestures and the system still breathes on its own.
- **No raw values, ever.** Touch input and audio are always smoothed before they reach a shader or a parameter — chaos is wild enough without feeding it jitter.
- **No WebGL fallback.** No WebGPU means a clean "unsupported" screen, not a degraded experience.

## Lorenz, 1963: deterministic chaos in three equations

A bit of history, because it makes the rest taste better.

**Edward Norton Lorenz** (1917–2008) was a meteorologist and mathematician at MIT. Starting from Barry Saltzman's equations for thermal convection, he reduced the problem to **three coupled ordinary differential equations** and published [*Deterministic Nonperiodic Flow*](https://journals.ametsoc.org/view/journals/atsc/20/2/1520-0469_1963_020_0130_dnf_2_0_co_2.xml) in 1963 — one of the most cited papers in the field.

The legend is real: two years earlier, re-running a weather simulation on a Royal McBee LGP-30, Lorenz re-entered a value as `0.506` instead of the stored `0.506127`. The forecast diverged completely. A rounding error in the fourth decimal had rewritten the weather. That is **sensitive dependence on initial conditions** — later nicknamed the **butterfly effect**, after his 1972 talk *"Does the flap of a butterfly's wings in Brazil set off a tornado in Texas?"* (The attractor even *looks* like a butterfly, which never hurt the branding.)

What makes the Lorenz attractor *strange*: the trajectory is **bounded** (it never escapes), **never periodic** (it never repeats), **never self-intersecting**, and it lives on a **fractal** of non-integer (Hausdorff) dimension ≈ 2.06. The system is **dissipative** — phase-space volumes contract at a constant rate, the divergence of the field being a negative constant — and yet trajectories are chaotic: everything collapses onto a zero-volume fractal, while two neighbouring paths pull apart exponentially. The rate of that pulling-apart has a number — the largest **Lyapunov exponent**, about **0.906** at the canonical parameters — and it is exactly how fast the butterfly effect compounds: a tiny error doubles roughly every three-quarters of a time unit, until all prediction is lost.

<Callout type="info" title="The parameters are physics, not magic numbers">
  In the convection model the three constants mean something concrete:

  - **σ (sigma)** is the **Prandtl number** — momentum diffusivity over thermal diffusivity.
  - **ρ (rho)** is a normalized **Rayleigh number** — essentially *how hard you are heating from below*. Below a critical value near 24.74 the flow settles; above it, chaos. The classic **ρ = 28** sits comfortably in the chaotic regime.
  - **β (beta)** is a geometric factor tied to the aspect ratio of the convection rolls.

  So the app's "intensity" knob is literally a thermostat. Fitting, for something born in a heat wave.
</Callout>

And you can *watch* a system cross into chaos by turning that thermostat. Hold σ and β at their canonical values and sweep ρ: at **ρ = 14** the flow spirals calmly into a fixed point and stays there; nudge it to **ρ ≈ 24**, just under the threshold, and it teeters — long, wandering transients that almost settle but don't; at **ρ = 28** it never settles again. Same three equations, one number hotter.

And it still matters. Lorenz's result is *why* weather is only predictable about two weeks out, and it is the reason modern forecasting is done as **ensembles** — run many slightly-perturbed initial states and forecast the *distribution*, not a single line. The same idea separates **weather** (a chaotic spell of heat) from **climate** (the baseline quietly shifting underneath). Beyond meteorology, the Lorenz system shows up in nonlinear electronics, population biology, secure communication via chaos synchronization, and a lot of generative art — including this one.

## The mathematics, brick by brick

### The system

$$
\frac{dx}{dt} = \sigma\,(y - x), \qquad
\frac{dy}{dt} = x\,(\rho - z) - y, \qquad
\frac{dz}{dt} = x\,y - \beta\,z
$$

With the canonical constants **σ = 10, ρ = 28, β = 8/3 ≈ 2.667**, this is a direct transcription:

```ts
function lorenz(p: Vec3, params: LorenzParams): Vec3 {
  const [x, y, z] = p;
  return [
    params.sigma * (y - x),
    x * (params.rho - z) - y,
    x * y - params.beta * z,
  ];
}
```

### Trick #1 — integrate the *normalized* field, so the ribbon flows at constant speed

Here is the first non-obvious decision. The raw Lorenz velocity varies enormously along the curve — fast on the outer sweeps, nearly stalled near the fixed points. If you generate ribbon geometry "per unit time", your tiles come out stretched in some places and bunched in others.

The fix: **normalize the field to unit length** and advance by **arc length** instead of by time.

$$
\hat{f}(\mathbf{p}) = \frac{f(\mathbf{p})}{\lVert f(\mathbf{p}) \rVert}
$$

```ts
function field(p: Vec3, params: LorenzParams): Vec3 {
  const v = lorenz(p, params);
  const len = Math.hypot(v[0], v[1], v[2]);
  if (len < FIELD_EPS) return v;          // don't divide by ~0 near fixed points
  return [v[0] / len, v[1] / len, v[2] / len];
}
```

Now the head walks the attractor at a steady pace no matter where it is, and a deposit dropped every fixed arc-length step produces a ribbon of perfectly even tiles. Speed becomes a clean multiplier on top, instead of a property of the geometry.

### Trick #2 — fourth-order Runge-Kutta (RK4), because chaos punishes sloppy integration

The shape only stays recognisable if the integrator is accurate, so the head advances with classical **fourth-order Runge–Kutta** over that normalized field:

$$
k_1 = \hat f(p),\quad
k_2 = \hat f\!\left(p + \tfrac{h}{2}k_1\right),\quad
k_3 = \hat f\!\left(p + \tfrac{h}{2}k_2\right),\quad
k_4 = \hat f(p + h\,k_3)
$$

$$
p_{n+1} = p_n + \frac{h}{6}\,(k_1 + 2k_2 + 2k_3 + k_4)
$$

There is no fixed wall-clock `dt`: the step `h` is an **arc-length step** (capped per sub-step), derived from the frame time times the current speed. Frame deltas are clamped so a stutter or a backgrounded tab can never blow the system up — and if a coordinate ever goes non-finite or the point wanders past a safe radius, it simply respawns near the origin.

### The "Clash" — two attractors and the butterfly effect, made visible

The name comes from running **two** attractors at once. Same `σ/ρ/β`, but **different initial conditions**: curve A starts near the origin, curve B starts somewhere else entirely. Because the system is chaotic, those two seeds **diverge** into two independent paths — the butterfly effect, staged as choreography. Curve B is then **rotated about 90°** so it physically crosses curve A at the center of the scene. That crossing is the *clash*.

### The hub — a ribbon that *records* a morphing chaos

Every frame, before the integrators step, a single function folds audio and motion into the live parameters, eased and clamped to stay in the "beautiful chaos" band:

$$
\sigma = \mathrm{clamp}\!\left(10 + \Delta\sigma_{\text{audio}} + 3.5\,t_x,\; 6,\; 16\right)
$$
$$
\rho = \mathrm{clamp}\!\left(28 + \Delta\rho_{\text{audio}} + 8.0\,t_y,\; 18,\; 42\right)
$$
$$
\beta = \mathrm{clamp}\!\left(\tfrac{8}{3} + \Delta\beta_{\text{audio}} + 0.35\,\text{twist},\; 1.8,\; 3.4\right)
$$

```ts
const sigma = clamp(10 + m.sigma + ms.tiltX * 3.5 * mm.tilt, 6, 16);
const rho = clamp(28 + m.rho + ms.tiltY * 8.0 * mm.tilt, 18, 42);
const beta = clamp(8 / 3 + m.beta + ms.twist * 0.35 * mm.twist, 1.8, 3.4);
this.lorenz.setParams(sigma, rho, beta);
this.lorenz2?.setParams(sigma, rho, beta);
```

Loudness and beats push **ρ** (the headline "chaos" knob), treble nudges **σ**, beats tap **β**; and your finger joins in — dragging across the artwork leans on σ/ρ and bends β. Because the parameters change *before* each deposit is laid down, the ribbon becomes a **recording**: its head is the chaos of *now*, its tail the chaos of a few seconds ago, frozen into geometry.

### A second use of Lorenz: dissolve masks

There is a second, completely separate Lorenz integrator that runs once at startup. It renders **ten 2D density textures** — projected "butterfly" portraits of the attractor — used as **dissolve masks**: full-screen background stills don't just fade in, they *materialise along the curves of the attractor*. Same mathematics, a totally different job. (This one uses plain semi-implicit Euler over tens of thousands of steps — accuracy doesn't matter for a static splat.)

## The graphics pipeline: WebGPU, hand-rolled

### Setup and the one limit that matters

Bring-up is standard: request an adapter and device, configure the canvas with the preferred format and `premultiplied` alpha, listen for device-lost and uncaptured errors (WebKit validates WGSL more strictly than Chrome — a silent validation error is a black frame on iOS). Every shader is compiled through a tiny wrapper that calls `getCompilationInfo()` and **throws on the first error message**, so a typo surfaces as text on a phone instead of nothing.

Only **one** optional limit is requested: `maxTextureArrayLayers`, pushed toward 1024. That number is the ceiling on ribbon length, because of the core idea below.

### The core idea: the ribbon *is* the integrator's history

The central processing unit (CPU) integrator drops a deposit every fixed arc-length step into a **ring buffer**. Each deposit is 12 floats — position, a birth sequence number (drives age), the tangent, a key-frame flag, and a **frozen snapshot of the audio** at the moment it was born. Each **camera frame** captured at a deposit becomes **one layer of a `texture_2d_array`** — hence the layer-count limit being the length limit. Because WebGPU external textures are frame-scoped, the renderer copies each deposit immediately into persistent storage instead of ever keeping a live camera reference.

The ribbon's vertex shader has **no vertex buffer at all**. Instance `s` reads two consecutive deposits straight from a read-only storage buffer and builds a camera-facing quad from `vertex_index`: the long axis follows the true 3D segment, the width axis is the screen-space perpendicular, sized so the tile never stretches.

```wgsl
let seg = B - A;
let segPlane = vec2<f32>(dot(seg, cam.right), dot(seg, cam.up));
let segLen = length(segPlane);
let dir = select(vec2<f32>(0.0, 1.0), segPlane / segLen, segLen > 1e-5);
let perp = -dir.y * cam.right + dir.x * cam.up;
```

The fragment shader cross-blends the two endpoints' camera frames from the texture array, and as a tile ages it earns the app's signature move — it **pixelates**, then each block **shrinks, drifts and spins** off into nothing, with the gaps discarded:

```wgsl
let prog = smoothstep(0.85, 1.0, age);
let rnd = hash22(cellId);
let drift = (rnd - vec2<f32>(0.5)) * prog * 0.5;
let ang = (rnd.x - 0.5) * prog * 1.8;
let cc = cos(ang);
let ss = sin(ang);
var lc = fract(baseUv * cells) - vec2<f32>(0.5) - drift;
lc = vec2<f32>(cc * lc.x - ss * lc.y, ss * lc.x + cc * lc.y);
let half = 0.5 * (1.0 - prog);
if (half <= 0.002 || any(abs(lc) > vec2<f32>(half))) {
  discard;
}
```

### Camera off? The renderer doesn't care

A key constraint was that the piece stays alive — and generative — with no camera. So the renderer is **source-agnostic**: a procedural `SyntheticSource` feeds the exact same array layers the camera would. It is a three-stage chain:

1. **Splat** — a persistent texture, faded slightly every frame, then additively inked with one soft blob per live Lorenz deposit. A self-portrait of the attractor, in feedback.
2. **Colourise** — the splat is sampled through a **domain warp** driven by 4-octave **fractional Brownian motion** (fBm), then mapped through an **Iñigo Quílez cosine palette**, the hue drifting off the live Lorenz `z`, the audio centroid and a slow low-frequency oscillator (LFO).

Here, fBm means "stack several layers of noise": each octave is twice as fine and half as strong as the previous one. One layer looks like smooth randomness; four layers start to feel like natural turbulence.

```wgsl
fn fbm(p: vec2<f32>) -> f32 {
  var s = 0.0;
  var amp = 0.5;
  var freq = 1.0;
  for (var i = 0; i < 4; i = i + 1) {
    s = s + amp * vnoise(p * freq);
    freq = freq * 2.0;
    amp = amp * 0.5;
  }
  return s;
}
fn pal(tIn: f32, shift: f32, sel: f32) -> vec3<f32> {
  let t = fract(tIn + shift);
  var d = vec3<f32>(0.0, 0.33, 0.67);
  if (sel > 1.5) {
    d = vec3<f32>(0.0, 0.10, 0.20);
  } else if (sel > 0.5) {
    d = vec3<f32>(0.3, 0.20, 0.20);
  }
  return vec3<f32>(0.5) + vec3<f32>(0.5) * cos(6.28318 * (t + d));
}
```

3. **Blit** — the result is copied into a ring layer, exactly like a camera frame.

Ribbon history, background dissolves and key-frame glows all keep working, oblivious to whether the matter came from a lens or from noise.

<Callout type="info" title="A note on lineage: the demoscene, in a browser tab">
  Hand-coded close to the metal, every pixel made by math instead of assets, synced to music in real time, the whole thing a flex of what the hardware can do — that is the **demoscene** ethos, the 1980s–90s subculture of Amiga and PC groups coaxing impossible real-time audiovisuals out of bare silicon. Lorenz Clash wears the lineage openly: the starfield is rebuilt procedurally (zero assets), the camera-off visuals are pure noise and feedback, and even the colour comes from **Iñigo Quílez's cosine palette** — IQ being a demoscene veteran whose `a + b·cos(2π(c·t + d))` trick is now standard procedural-graphics vocabulary. Swap the Amiga for a phone GPU and the assembly for WGSL, and the spirit is the same.
</Callout>

### The frame, pass by pass

Everything happens inside one command encoder, one submit. Almost all of the visual work is render pipelines — every particle effect is computed analytically in a vertex shader over instanced cubes. The only compute pass in the shipped app is a tiny iOS camera-copy probe that scans for pure-black edge bands before the frame is stored. The order:

1. **Capture passes** — write the camera (or synthetic) frames into ring-buffer layers. Must come first, since the scene samples them.
2. **Trail passes** — a half-res persistent buffer is faded and re-inked, for smoke-like smear.
3. **Scene pass** (depth-tested, into an offscreen target when bloom is on): starfield dome → deformation grid → ribbons and morphing shapes (curve A, then curve B rotated) → comets and their wakes → tail-dissolution particles → the head crest.
4. **Bloom** — the post chain.
5. **Blit** — present.

A quick tour of what lives in that scene pass:

- **Travelers / comets** — a cube and a sphere ride each curve along a Catmull-Rom spline, spiralling in the plane perpendicular to the tangent (in phase opposition), trailing a helical **wake**.
- **Shape morph** — a head-to-tail wave that turns the ribbon into cubes, then spheres, as a *geometric* cross-fade rather than an alpha blend (the incoming form grows from the head while the outgoing shrinks).
- **Starfield** — a procedural dome reconstructed from the camera basis, so the stars rotate correctly as the camera orbits. Zero assets.
- **Head crest** — additive cubes riding the newest deposit on an audio-reactive sine wave: the visible "now" of the ribbon.
- **Deformation grid** — a fBm-displaced floor lattice, normally subtle, but sometimes hit by accelerating rainbow wave fronts on rhythmic cues.
- **Key-frame glow** — when a tile corresponds to a captured high-definition (HD) still, its deposit carries a flag and pulses with a desynchronised colour accent.

That grid is worth unpacking because it is not just a horizon line. It is the one place where the invisible control field becomes directly visible: a regular 24×24 line lattice samples an animated fBm height field, then the smoothed pointer gesture leans that surface like a tray.

```wgsl
let n = fbm(p.xz * noiseScale + vec2<f32>(t * flow, t * flow * 0.7)) - 0.5;
world.y = world.y + n * noiseAmp * amp;
world.y = world.y + (g.field.x * p.x + g.field.y * p.z);
```

The occasional rainbow burst is a second field layered on top. Several radial sine crests expand from the centre, accelerate as `phase²(1 + 2.2phase)`, get sharpened through a power curve, and fade through a smooth envelope:

```wgsl
let accel = phase * phase * (1.0 + phase * 2.2) * rainbowSpeed;
let crestPhase = radial * waves - accel * (waves + 1.5) + seed;
let crest = pow(0.5 + 0.5 * sin(crestPhase * 6.2831853), sharpness);
```

While that event runs, the renderer also lifts the grid opacity toward 100%, so the floor briefly becomes an emissive waveform before dissolving back into the quiet blue lattice.

<Callout type="info" title="War story: eight uniform buffers, because writes outrun draws">
  The comets are drawn **eight** times a frame — two curves, each with a cube body, a sphere body, and a wake trailing each of those. The tidy version reuses one small uniform buffer and rewrites it before every draw. It renders all eight comets *identically* — and the reason is queue ordering: `queue.writeBuffer` doesn't run where you call it, it is enqueued, and the whole frame is one command encoder with a **single `submit`**. So all eight writes execute *before* any draw reads, and every draw reads the **last** payload written. The fix is one uniform buffer per simultaneous draw, each written immediately before its own draw call:

  ```ts
  const cometUniforms = Array.from({ length: 8 }, () =>
    device.createBuffer({
      size: 20 * Float32Array.BYTES_PER_ELEMENT,
      usage: GPUBufferUsage.UNIFORM | GPUBufferUsage.COPY_DST,
    }),
  );
  this.writeCometUniform(0, this.cometA, 0, count, 0, 0, time, sA);
  pass.setBindGroup(0, this.cometBindGroups[0]);
  pass.draw(this.cubeVertexCount, 1);
  this.writeCometUniform(1, this.cometA, 0, count, 0, Math.PI, time, sA);
  pass.setBindGroup(0, this.cometBindGroups[1]);
  pass.draw(this.sphereVertexCount, 1);
  ```
</Callout>

### The camera frames itself

None of that matters if the viewpoint can't hold it. By default the camera orbits slowly at a 50° field of view, but it also **auto-frames**: each frame it takes the bounding box of every live deposit (both curves) and pulls back just far enough to fit that bounding sphere into the **tighter** of the two field-of-view axes.

```ts
const halfV = FOV_Y / 2;
const halfH = Math.atan(Math.tan(halfV) * aspect);
// fit into the tighter axis → a tall phone steps back, a wide screen leans in
const dist = (frameRadius * margin) / Math.sin(Math.min(halfV, halfH));
```

A portrait phone steps back to keep the whole clash in shot; a wide desktop leans in to fill the frame. Centre and radius are eased toward their targets at a crawl — a smoothing factor of `0.03` — so the framing *drifts* as the attractor breathes rather than pumping on every wobble. And the fit deliberately uses the **base** field of view (FOV), so a pinch-to-zoom (applied as an FOV scale, `FOV_Y / zoom`, a genuine telephoto) still works *on top* of the auto-frame instead of being cancelled by it.

Two **follow** modes trade the orbit for a ride along the curve. *Head* sits just behind the newest deposit and looks ahead down the smoothed tangent — first-person chaos. *Tail* sits at the oldest, dissolving end and looks back at it — except the curve coils, so the local tangent spins a full turn inside every loop and would send the eye pirouetting. The fix is to aim not along the tangent but at an **actual deposit further up the trail**, using the chord toward it as a stable bearing, so the dissolving tail stays framed even through a loop.

### Compositing and post

Bloom is a half-res ping-pong: a soft-knee **bright pass** that preserves hue, a **5×5 box blur run twice** (a cheap, slightly-wide gaussian), then an **additive composite** back over the scene. Additive on purpose — only the bright, emissive elements (head crest, edge tints, comets, key-frame glow) bloom; the camera material stays clean, so the glow reads as localized light rather than a global haze. The composite also hides the app's small lens tricks: a radial **ripple** that warps the sample during slow-motion, a second timed **pixel ripple** fired by rhythmic cues, and Poisson-scheduled **camera-flash** pops that briefly dip the threshold and spike the intensity.

The ripple deserves a closer look, because it is pure shader math and costs nothing extra. Pull time into slow motion and the composite pass bends the whole image with a **radial travelling wave**. For each pixel it takes the vector from screen centre $\mathbf{c} = \mathbf{uv} - 0.5$, its length $d = \lVert \mathbf{c} \rVert$, and pushes the *sampling* coordinate outward along that radius by two blended sine waves of the distance:

$$
\mathbf{uv}' = \mathbf{uv} + \frac{\mathbf{c}}{d}\,A\,[\,\sin(kd - 2t) + 0.35\,\sin(0.53kd + 1.35t)\,]
$$

```wgsl
let c = uv - vec2<f32>(0.5);
let d = length(c) + 1e-5;
let w = (
  sin(d * bloom.fx.z - bloom.fx.y * 2.0) +
  sin(d * bloom.fx.z * 0.53 + bloom.fx.y * 1.35) * 0.35
) * amp;
uv = uv + (c / d) * w;
let scene = textureSampleLevel(tex, samp, uv, 0.0).rgb;
let glow = textureSampleLevel(bloomTex, samp, uv, 0.0).rgb;
```

A handful of choices keep it tasteful rather than gimmicky:

- It warps the *sample* coordinate — an inverse-mapped lens — so there is no geometry and no extra pass; concentric rings expand from the centre as the $-2t$ term slides the sine outward, like ripples from a dropped stone.
- The spatial frequency is fixed ($k = 16$, about one or two visible rings) and the amplitude is tiny — $A = \text{slowmo} \times (0.012 + 0.01\,\text{beat})$, at most ~2% of the screen — so it *suggests* reality bending without making you seasick.
- It is gated on a non-zero amplitude, so at normal speed the branch is skipped and the effect is literally free.
- Scene **and** glow are sampled with the *same* warped `uv`, so the bloom ripples in lock-step instead of detaching from the image.
- The phase $t$ is the **warp-scaled** scene clock, so the wave slows with the music and eases back on release — the ripple belongs to the same tape-stop gesture, not a separate animation.

That slow-motion ripple later became its own effect. The **pixel ripple** uses the same composite pass, but with a separate event uniform: when a roll, snare or strong kick lands, the app launches a short radial wave with randomized amount, duration, frequency and speed. The randomness matters. A fixed ripple looks like a canned transition after three plays; a bounded range makes it feel like the music is throwing pressure waves through the scene. If the score is quiet, a sparse Poisson fallback can still trigger one, so the effect is music-timed without depending on the auto-director being enabled.

The more expensive-looking effect is actually the simplest one conceptually: an **exposure trail**, like leaving a camera shutter open for a few seconds. After the final composite, a tiny temporal-feedback pass mixes the current image with a decayed copy of the previous output in two owned ping-pong textures. It fades in, holds, fades out, and never reads the live swapchain. There are two timings: short hits of about **1.5 s**, usually around cuts or strong rhythmic events, and long trails of about **20 s** for calmer floats, bridges and breakdowns. The counts scale with song length: at four minutes the cap is **20 short** and **3 long** plays, with shorter or longer tracks scaled proportionally. Like the ripples, it is score-driven but independent of the camera director — turning off auto-camera should not turn off the visual life of the music.

It is a good miniature of the whole compositing philosophy: fold the effect into the pass that already touches every pixel, or into a single owned post pass when temporal history is needed, and keep the renderer's layer count under control.

<Callout type="info" title="War story: never read the live swapchain on iOS">
  The final image is rendered into an **owned** texture and then blitted to the canvas. Reading from `getCurrentTexture()` directly — the obvious thing — stalls and corrupts presentation on iOS Safari. The owned-texture detour also gives video export a stable surface to grab, without ever touching the live swapchain.
</Callout>

### The audio-reactive layer

Audio is an `AnalyserNode` with a Fast Fourier transform (FFT) size of 2048, tapped **before** the master gain, so muting the music silences your ears but keeps the visuals dancing. Each frame it extracts level as root mean square (RMS), bass/mid/treble (mean linear amplitude over Hz bands, normalized against offline-measured percentiles), a log-mapped spectral **centroid**, and a **beat** estimate (bass flux above a slow running average — fast attack, slow decay).

Those features drive three separate parts of the system:

- **Live and global**, packed into the shared camera uniform — driving the head crest, edge tints, bloom flashes and parameter modulation right now.
- **Frozen per deposit** — the snapshot baked into each ribbon tile at birth. This is what makes the ribbon a recording: each tile keeps the **color** (centroid → palette, on the edges only), **size** (bass + beat → grow) and **spin** (treble) of the instant it was created, even as the music moves on.
- **Transport time** — the slow-mo gesture produces a smoothed `warp` from 1.0 down to about 0.35. That same number slows the music, the Lorenz scroll, the deposit rate, the comet motion and the composite ripple, so the tape-stop reads as one event instead of five unrelated effects.

The mapping is deliberately small enough to stay legible. **Level plus beat** push **ρ** and scroll speed, which makes the attractor feel like it is breathing with the kick. **Treble** nudges **σ** and adds spin/shimmer. **Beat** taps **β** and the bloom flashes. **Centroid** chooses the palette, but mostly on the edges so the camera texture in the middle stays readable.

### From FFT to a score: offline music structure

The live FFT is good at *now*: how loud, how bright, how much bass, whether something just attacked. It is not good at *where we are in the song*. An intro, a breakdown, a drive section and a floating bridge can have similar instantaneous energy for a few seconds, but they ask for different camera language. An intro wants a wide orbit. A driving section can tolerate follow shots, rapid cuts and comets. A calm bridge is where a long exposure trail feels earned.

So the music has a second layer: an **offline score** generated once with `librosa`, committed as JavaScript Object Notation (JSON), and loaded by the app at runtime. The script is a self-contained `uv` Python script:

```bash
uv run --python 3.12 scripts/analyze-track-structure.py
```

For each bundled `.m4a`, it resamples to **22.05 kHz**, computes onset strength, tempo and beat times, then derives RMS, spectral centroid, chroma and Mel-frequency cepstral coefficient (MFCC) features on a one-second grid. A novelty curve compares the local feature profile before and after each time point (an eight-second context on each side), peaks become candidate section boundaries, and each section gets a coarse role: `intro`, `drive`, `groove`, `break`, `bridge`, `float` or `outro`.

The same pass extracts rhythmic cues in separate bands: low-band flux for **kicks**, high-band flux for **snares**, mid-band flux for **toms**, and short dense snare sequences for **rolls**. Across the 11 tracks, the current JSON contains **69 sections**, **4,875 beats**, **2,966 kicks**, **2,998 snares**, **1,408 toms** and **115 rolls** — about **42 minutes** of pre-analysed timing, still with no server and no runtime machine learning (ML).

The runtime then treats that JSON less like "analysis" and more like a score:

```json
{
  "start": 186,
  "end": 238.379,
  "role": "drive",
  "energy": 0.583,
  "rhythm": 0.222,
  "brightness": 0.76,
  "director": { "view": "followA", "shape": "cube", "comet": true }
}
```

There are two directors now. The original one is still there: a classic live-FFT novelty detector that compares a short-term profile against a long-term one and switches when the sound drifts far enough. The structured director sits beside it, unchanged by the live detector. When playback enters a new analysed section, it applies the section's preset; when beats, snares, toms and rolls pass the playhead, it emits small cues that can trigger **zoom pulses**, **rapid camera jumps**, **comets**, **grid waves**, **pixel ripples** and **exposure trails**. The colour, grow and shimmer still come from the live FFT, so the score never freezes the performance — it just gives the system a map.

That separation matters. You can turn the camera director off and still keep the score-driven effects: the music can keep throwing ripples and exposure moments at the renderer without taking over your view. It also keeps the experience from feeling like a fixed music video. Section changes are deterministic anchors, but the micro-events keep probability, cooldowns and random ranges, so the same track has a recognisable structure without replaying the exact same cuts every time.

<Callout type="info" title="War story: iOS broke the obvious slow-mo">
  The obvious implementation is `audioElement.playbackRate = warp`. It works until that element is routed through `createMediaElementSource()` on iOS, where changing the rate can cut or re-buffer the sound. The workaround is heavier but stable: decode the current track ahead of time, switch to an `AudioBufferSourceNode` when `warp < 0.99`, drive that node's `playbackRate`, and feed it into the same analyser. Returning to normal is just `rate → 1` — no seek, no restart. The original `<audio>` element only takes over again on the next track.
</Callout>

The genre earns its place — its steady four-on-the-floor kick gives the bass-flux beat detector something unambiguous to lock onto, and its long builds and drops map naturally onto ρ swelling and easing back.

## Recording it: local video export, no server

The piece is generative and never repeats, so the next wish is obvious: to *keep* a moment of it. Lorenz Clash records and exports video — a real `.mp4`, with the music muxed in — and, true to the on-device rule, **none of it touches a server**. It all runs in the browser, on the phone, which is exactly where it gets interesting: encoding video in a tab without running memory off a cliff.

### Where the frames come from

Capture reuses the owned-texture trick from the compositing war story. A recorded frame is **never** read from the live canvas; instead the finished composite — already sitting in the owned `presentTexture` — is blitted into a separate `OffscreenCanvas` (its own WebGPU context), and that canvas is wrapped as a WebCodecs `VideoFrame`:

```ts
// blit the OWNED present texture into an offscreen capture canvas, then snapshot it
pass.setPipeline(this.blitPipeline);
pass.setBindGroup(0, this.blitBind);
pass.draw(3);
this.device.queue.submit([encoder.finish()]);
return new VideoFrame(this.captureCanvas, { timestamp: timestampUs, duration: durationUs });
```

Reading the live swapchain per frame is what glitches iOS Safari — the export path never does it, and the same owned canvas backs the PNG screenshot path too.

### Streaming encode: hold chunks, never frames

A naïve recorder hoards raw frames and encodes at the end — but a minute of 1080p raw frames is gigabytes. Instead each frame is fed straight to a WebCodecs `VideoEncoder` at 30 fps and **closed immediately**, so RAM only ever holds the *compressed* output (a few MB per clip):

```ts
const frame = this.src.grabCompositeFrame(this.outW, this.outH, tsUs, FRAME_DUR_US); // tsUs: real elapsed µs
this.encoder.encode(frame, { keyFrame: s.frameIndex === 0 }); // one shared encoder, reset per clip
frame.close(); // critical: never accumulate raw frames
```

The hardware encoder already runs off-thread, so no Web Worker is needed — the main thread is barely involved. Codec choice is a quick capability probe: H.264 level 4.0 first (mobile), falling through to 5.0/5.1/5.2 for big desktop/Retina frames that blow past level 4.0's macroblock ceiling; if no H.264 encoder exists at all, the whole feature quietly disables. A *clip* (a "sequence") is one background appearance — an HD still dissolving in, then out — captured automatically, plus one-tap manual snapshots, collected for you to curate (see *Surviving it* for why auto-capture pauses rather than recycling).

<Callout type="info" title="War story: encoded chunks live on disk, or mobile OOMs">
  Even compressed, video adds up — roughly 1 MB per second — and a long session of snapshots would pin all of it in the JavaScript (JS) heap and crash the tab on a phone with an out-of-memory (OOM) failure. So the encoded H.264 chunks are offloaded to **IndexedDB**: RAM keeps only light metadata and a 200 px thumbnail per clip; the bytes live on disk. The store is **session-scoped** — wiped on every launch — which also means camera-derived video never lingers on the device after you leave. No IndexedDB (private mode)? It falls back to keeping chunks in RAM. Storage pressure is read from `navigator.storage.estimate()`: automatic clips stop accumulating past 60% usage, manual snapshots past 85%.
</Callout>

### Exporting: mux, don't re-encode

Because every frame was already encoded *live*, assembling the file is cheap — it is a **mux**, not a re-encode. [mp4-muxer](https://www.npmjs.com/package/mp4-muxer) stitches the clips by re-basing each clip's timestamps so they play back-to-back, pulling each clip's chunks from IndexedDB **one clip at a time** so the heap never refills with the whole reel:

```ts
for (const chunk of payload.chunks) {
  muxer.addVideoChunk(chunk, metaWritten ? undefined : payload.meta, chunk.timestamp + tOffsetUs);
  metaWritten = true;
}
tOffsetUs += seq.durationUs; // next clip starts where this one ended
```

### The soundtrack is the real music

The audio in the export isn't a recording of the output — it's the **actual track** that was playing, re-decoded from the local `.m4a`, sliced to each clip's window and laid end-to-end with ~10 ms boundary fades so the joins don't click. Two small war stories made this harder than it looks.

<Callout type="info" title="War story: decode with an OfflineAudioContext, never a real one">
  Decoding the track needs an `AudioContext` — but creating a *real* one grabs the hardware audio session, and on iOS that **suspends the live music's context**, silencing the app until a reload; every export or preview tap would kill the sound. The fix is to decode through an **`OfflineAudioContext`**, which renders to a buffer and owns no output session, so it is completely inert. And since decoding a full multi-minute track is by far the heaviest step, the most-recently-decoded track is cached — but only **one** entry, because a decoded 8-minute stereo track at 48 kHz is ~180 MB and holding two would OOM.
</Callout>

When you pull the visuals into slow-motion, the sound has to slow with them — so each recorded frame stores its slow-mo `warp`, and the export **resamples the music along that envelope**: the read head advances more slowly through the samples exactly where the video was tape-stopped, with linear interpolation of the now-fractional position. One last trap: mobile `AudioEncoder`s often omit the AAC `AudioSpecificConfig`, and without it mp4-muxer writes a broken `esds` box and the file plays silent — so that 2-byte config is synthesized by hand and handed to the muxer explicitly.

The result is a single self-contained `.mp4` — your slice of the chaos, with its music, made entirely on the phone.

## Surviving it: the iPad that ran out of GPU

It ran beautifully on an iPhone 13 and an iPhone SE — then crashed an **iPad Pro 2018** after a few minutes. The reflex is "old device, less memory," but that iPad has the *same* 4 GB as the phones, or more. What it has that they don't is a much **bigger screen** and, that day, the camera held in **landscape**. Both mattered, and chasing why is the most instructive part of the build.

Safari's remote Web Inspector was useful while the page was alive, but it was not enough for the failure itself: once the GPU process disappeared, the useful state had to already be visible on the device. So the app grew a **diagnostics line** — printed every ten seconds and *baked onto the crash screen itself* — reporting frame rate, clip count, encoder count and the total bytes of every texture it had allocated. A screenshot of the crash became a full readout.

<Callout type="info" title="War story: it ran out of GPU memory, not RAM — and iOS won't say">
  The crash was always a GPU allocation failing — `beginRenderPass` or `importExternalTexture` returning nothing — never a JavaScript out-of-memory. WebKit's **GPU process** has its own budget, separate from the page heap, and a big screen plus a live camera fills it fast. Worse, **iOS exposes no memory API** at all (`deviceMemory`, `performance.memory`, `measureUserAgentSpecificMemory` are Chrome-only) and WebGPU reports no video memory (VRAM) — you fly blind. And once that process dies, `requestAdapter()` returns `null`: the whole subsystem is gone, so there is no recovering in place, only a reload.
</Callout>

The biggest single lever was the **ring buffer**. Its layers are sized to the camera's aspect ratio, and a landscape frame holds about **three times** the pixels of a portrait one — so on the landscape iPad the ring alone was **248 of 288 MB**. Capping each layer's *longer* edge instead of its height makes the footprint orientation-independent: 288 MB became **125 MB**, and the already-small portrait phones were untouched.

<Callout type="info" title="War story: one encoder, reset between clips">
  The recorder first built a fresh `VideoEncoder` per clip. On iOS that churns **VideoToolbox sessions**, and after a dozen or two the next one fails and takes the GPU process with it. The fix is **one** encoder for the whole session, `reset()` between clips — which quietly fixed two more bugs: reusing an encoder across clips with each clip's timestamps restarting at zero had been feeding it *backward* time jumps (corrupt, frozen frames), and a shared encoder emits its `decoderConfig` only once, so every clip after the first was muxed with none and wouldn't open. A clean reset makes each clip an independent, well-formed encode.
</Callout>

The leak that set the ceiling was the camera. `importExternalTexture()` is minted fresh on demand and iOS reclaims those transient external textures slowly. The obvious fix — copy the frame once into a texture I own and stop importing — is exactly what worked in the end, but it first broke in the most iOS way possible.

<Callout type="info" title="War story: the black rectangle detector, then the square stream">
  Swapping `importExternalTexture` for `copyExternalImageToTexture` into an owned texture removed the slow iPad leak, then rendered the iPhone camera as a stretched image with a pure-black side rectangle. Hard-coding a crop fixed one phone orientation and broke another. The first real breakthrough was a tiny GPU probe: scan the copied frame at full resolution, count edge rows and columns that are pure black, ignore all-black startup frames, take three samples, then crop and recompute the visible aspect ratio before the ribbon stores the image.

  The final fix was even better: on Apple mobile, request the camera as a **square stream** (`720×720`, aspect `1`). Lorenz Clash uses the camera as square material anyway, and this nudges WebKit into returning a centered, already-usable frame. The black-rectangle detector stays in place as a diagnostic and fallback, but the happy path now logs `track 720x720 aspect=1.000` and `copyCrop=none`. iOS/iPadOS use this owned copy path; desktop and Android keep the simpler external-texture path.
</Callout>

There was also a cheap win before that: **only touch the camera on frames that actually capture** a ribbon deposit. Deposits arrive irregularly, so the in-between frames had been minting textures for nothing. That reduced the leak while the copy-path work was still unstable, and it remains useful because it lowers camera work on every device.

Export, on a tight device, is both victim and trigger — and the two fixes are the same idea twice: **react, don't measure** (you can't measure). With no memory API to size the export to the device, the encoder's own **faults** are the signal — each failure drops the export resolution a notch and retries, so a capable machine stays at full resolution while a GPU-tight one ratchets down to something sustainable. And a device that can't hold 30 fps during capture drops frames: timestamping by frame *count* exported a 5-second take as a shorter, sped-up clip, so frames are now stamped by **real elapsed time**, keeping the duration honest at the device's true rate.

The last fix was a policy. Background clips had been a *rolling buffer* — evict the oldest at the cap, keep encoding — which means the hardware encoder runs **continuously** for as long as the app is open, and that sustained load is what finally tips a tight device over. So auto-capture now **pauses** at the cap instead, and a small curation user interface (UI) — keep, trash, or "clear unkept" — frees slots to resume. Don't encode forever just to throw it away.

The result: continuous camera-and-capture went from a crash at **~5 minutes** to **past thirty** on the same iPad, GPU memory flat throughout. The final iOS copy path now logs the square camera stream (`track 720x720 aspect=1.000`) and removes the `importExternalTexture` counter from the long run (`camImp=0`, `camCopy` grows at the deposit rate). The iPad can still get hot — this is a 2018 tablet running WebGPU, camera and video encoding in a browser — but the failure mode changed from a slow GPU-memory cliff to ordinary thermal/performance pressure. That is the difference between a demo that dies and an experience I can ship.

## Shipping it: a fullscreen PWA

A **Progressive Web App** (PWA) is a website that the browser can treat more like an app: it can be added to the home screen, get its own icon, launch in a standalone or fullscreen window, and cache enough of itself to open reliably. For Lorenz Clash, the important part is not offline mode; it is **immersion**. The piece needs to escape the browser chrome, especially on mobile, so the camera canvas can fill the screen like an installed app.

That installable shape comes from two small web primitives: a **web app manifest**, which describes the app's name, icons, colors and display mode; and a **service worker**, which lets the browser recognise that the site has an app shell it can control. Chrome on desktop and Android supports the full install flow from the browser UI. On iOS and iPadOS, install it from Safari: open the site, tap Share, then **Add to Home Screen**.

I kept those two files explicit rather than hiding them behind a generated setup: the manifest and the service worker are hand-written static files.

`display: "fullscreen"` is the most chromeless mode, with `standalone` as a graceful fallback. Theme and background are the same near-black as the scene, so the launch screen has no seam. The 192 and 512 icons are each declared twice — `any` and `maskable` — for Android's adaptive icon.

### The service worker

Its stated job is blunt: **make the app installable**. Android Chrome only offers "Install" (and a standalone launch) when a service worker with a `fetch` handler is registered over HTTPS. The caching strategy is deliberately minimal and dev-safe:

```js
// navigations: network-first, fall back to the cached shell offline
if (request.mode === "navigate") {
  event.respondWith(
    fetch(request).catch(() => caches.match("/index.html").then((r) => r ?? caches.match("/"))),
  );
  return;
}
// hashed build assets: cache-first (content-addressed, safe forever)
if (url.pathname.startsWith("/assets/")) {
  event.respondWith(
    caches.match(request).then(
      (hit) =>
        hit ??
        fetch(request).then((res) => {
          const copy = res.clone();
          void caches.open(CACHE).then((cache) => cache.put(request, copy));
          return res;
        }),
    ),
  );
  return;
}
// audio tracks, dev modules, everything else → straight to the network
```

The large audio files are intentionally *not* cached, and because only content-hashed `/assets/` paths are ever cached, the dev server's HMR is never intercepted. The shell is installed with `allSettled` (not `addAll`) so one missing file can't fail the whole install.

### Fullscreen on iOS, where the bodies are buried

iOS ignores the manifest's `display` field, so standalone mode comes from the old Apple metas — `apple-mobile-web-app-capable`, a `black-translucent` status bar — plus `viewport-fit=cover` to extend under the notch. But the two decisions that actually cost real debugging time:

```css
#gpu-canvas {
  display: block;
  width: 100vw;
  height: 100dvh;   /* dynamic viewport — not 100vh */
  touch-action: none;
}
/* Deliberately NOT position: fixed. A fixed GPU canvas gets its own
   compositing layer that iOS Safari fails to repaint → black canvas. */
```

<Callout type="info" title="War story: camera orientation is not a reliable API">
  Camera metadata is not a reliable contract on mobile Safari. Before the square-stream copy path, iOS could report **portrait pixel dimensions for a physically landscape camera buffer**; on other feeds, width and height can tell you the aspect but not always the final browser-applied orientation. So the app keeps a **user-calibrated** boolean, persisted to `localStorage`: a "Rotate / Aspect" toggle on the start screen, previewed in what-you-see-is-what-you-get (WYSIWYG) mode through the exact same capture path the renderer uses. From it, the app derives a 90° auto-rotation and inverts the aspect on a quarter turn, so the image is never stretched on any platform.

  ```ts
  const screenLandscape = window.innerWidth >= window.innerHeight;
  const auto = bufferLandscape === screenLandscape ? 0 : 1; // disagree → +90°
  const rotation = (auto + Math.round(this.captureParams.rotation)) & 3;
  const odd = rotation === 1 || rotation === 3;
  const bufferAspect = bufferLandscape ? mag : 1 / mag;
  const aspect = odd ? 1 / bufferAspect : bufferAspect;
  ```
</Callout>

Safe-area insets (`max(<fallback>, env(safe-area-inset-*))`) keep the floating UI clear of the notch and the home indicator. On Android, Chrome surfaces its own install prompt once the criteria are met; on iOS it is the manual *Share ▸ Add to Home Screen*. No in-app coaching, no `beforeinstallprompt` — install is left to the browser.

### Getting out of the way

You start on a small home screen: choose camera and music, switch language if needed, then tap to begin. The camera preview is already rendered through the same capture path that will feed the ribbon, so orientation fixes happen before the experience starts, with no second permission prompt.

Inside, the controls stay minimal so the piece — not the chrome — owns the screen:

- **They hide.** Touch or move and the UI fades in, then disappears again after a few seconds.
- **They're ranked by the thumb.** Four diagonal zones group sound, source, view and capture; the main action is largest and lowest, supporting actions climb away from it.
- **Colour means state.** Buttons stay plain white; active modes are shown by rings, including the rainbow auto-director ring and the red capture progress ring.

The language selector lives only on the home screen, and the sparse in-run icons are explained by a paged Help legend that mirrors the button layout. Enough scaffolding to stay usable, then it gets out of the way.

## Closing the loop

A heat wave made me stare at rising air; rising air is convection; convection is what Lorenz reduced to three equations in 1963; those equations are chaos; and chaos, rendered with a camera and a phone GPU, turned out to be beautiful. Deterministic but unpredictable, bounded but never repeating — the attractor is a good metaphor for a hot summer that you can feel but never quite forecast.

And when you point the camera at your own face, the metaphor closes on you. You are *part of* the system — you put yourself into it, you are responsible for what it has to work with — and yet it runs away from you, because that is what chaos does. Which is, more or less, our exact relationship with the climate that made this summer so hot: we are inside it, we shape it, and it still exceeds our grasp.

If you read this far, go run it during the next heat wave — and let it turn your own reflection into the shimmer.

Dartvex is a set of Dart and Flutter packages for using [Convex](https://convex.dev) from the Dart ecosystem.

The goal is simple: make Convex usable in Dart and Flutter projects without waiting for an official SDK. The package family covers the main needs of a real app: realtime subscriptions, queries, mutations, actions, Flutter integration, typed bindings, authentication and offline-friendly patterns.

Dartvex is not an official Convex project. It is an independent release, published on pub.dev under my verified publisher account.

## Why Dartvex exists

Convex already has strong first-party support for JavaScript and React environments. Flutter teams, however, often need the same kind of realtime backend experience from Dart.

That demand is visible on the Convex public roadmap: the request for an [official Flutter/Dart SDK](https://ship.convex.dev/requests/js7dcrtm07s6ysfk7t1m1w5tfs849t11) is heavily backed, and it asks for a pure Dart client, realtime subscriptions, authentication, optimistic updates, offline-friendly patterns and code generation.

Convex has also indicated in that same request that a third official implementation of the client-side state machine is extremely unlikely. In practice, that means Flutter and Dart developers should not expect a first-party Dart SDK to appear.

Dartvex exists for that gap: a Dart-native ecosystem for teams that want Convex in Flutter or Dart today.

## The package ecosystem

The first public release was published on June 19, 2026. The current ecosystem is split into five packages, so developers can install only the layers they need.

| Package | Role |
|---|---|

| [dartvex_flutter](https://pub.dev/packages/dartvex_flutter) | Flutter integration layer with provider-style setup and reactive widgets for queries and mutations. |

## How to choose the packages

For a Flutter app, the usual starting point is [dartvex_flutter](https://pub.dev/packages/dartvex_flutter). It brings the UI layer and depends on the core [dartvex](https://pub.dev/packages/dartvex) package.

For a pure Dart application, command-line tool or server-side Dart project, [dartvex](https://pub.dev/packages/dartvex) is the base package.

When the backend schema should become a typed Dart API, [dartvex_codegen](https://pub.dev/packages/dartvex_codegen) adds the generation step.

When the product needs to keep working with unstable connectivity, [dartvex_local](https://pub.dev/packages/dartvex_local) adds the local-first layer.

You have decided to build a mobile app. It is often a major growth lever. But very quickly, one question appears: **which technology should you use?**

Capacitor. React Native. Flutter. Native. Four names that come up in every proposal, every discussion with a developer, every technical article. And yet most decision-makers have no concrete idea what these terms mean, or what they imply for budget, timeline and final product quality.

This article is here for that. No unnecessary jargon, no framework wars between developers. Just the **real trade-offs**, explained clearly.

## A choice that is both technical and strategic

Before talking about frameworks, ask the right questions. The technology choice will determine four critical variables for your company:

- **Your time-to-market**: how long before you are live on the stores?
- **Your total budget**: at launch, but also every year for maintenance.
- **Perceived quality**: smoothness, animations, respect for iOS and Android conventions. This often determines whether a user keeps the app or deletes it after 30 seconds.
- **Your ability to adapt**: will you be able to integrate new AI features, future sensors and tomorrow's system APIs?

The answers to these questions guide the choice far better than any performance benchmark. But make no mistake: it is also a technical choice. The selected technology has direct consequences on what you will, and will not, be able to do with your application.

## The four approaches, explained simply

<ThemeImage
  lightSrc="/images/2026/appexpress/quelle-technologie-mobile-choisir/architectures.webp"
  darkSrc="/images/2026/appexpress/quelle-technologie-mobile-choisir/architectures.webp"
  alt="The four mobile architectures compared"
  width={1600}
  height={893}
/>

### Capacitor (Ionic): the web inside a native shell

Capacitor uses **web technologies** (HTML, CSS, JavaScript) to build the mobile interface, displayed inside an embedded browser within the app. That does not mean putting a website inside an app. You build a dedicated mobile interface, with its own components and UX conventions, using Ionic or other mobile UI frameworks. When you need access to a phone feature, such as the camera, notifications or GPS, a plugin system bridges the web layer and the device hardware.

**Advantages:**

- Very fast development: web developers (JavaScript, React, Vue, Angular) are productive immediately.
- One codebase for iOS, Android and the web.
- Access to a full ecosystem of mobile UI components (Ionic, Konsta UI) that respect iOS and Android conventions.
- Ideal for content apps, internal tools and dashboards.

**Limits:**

- The interface remains "mobile web": it can lack the smoothness and animations that iPhone or Android users expect from a native app.
- Complex interactions, such as drag-and-drop, fluid animations and transitions, are harder to polish.
- If you need advanced hardware features, you will need to write native plugins, which requires additional skills.

**In one sentence:** the most accessible option for a web team, with a real mobile rendering, but limited when you aim for a premium experience.

### React Native: the popular compromise

React Native works differently: your code is written in JavaScript, the language of the web, but the interface displayed on screen uses the phone's **real native components**. A React Native button is a real iOS or Android button, not a web imitation.

In 2026, React Native has completed a major overhaul of its internal architecture. Communication between your code and the phone has become much faster and smoother. The micro-stutters that could be seen a few years ago have largely disappeared.

**Advantages:**

- One codebase for iOS and Android, with a truly native rendering.
- The largest pool of available developers, especially in France.
- The Meta ecosystem, Facebook, Instagram and WhatsApp, uses it in production, which supports its long-term viability.
- Ability to integrate React Native screens into an existing native app.

**Limits:**

- The toolchain is complex and evolves quickly: you need a developer who follows the ecosystem.
- Third-party libraries must be compatible with the new architecture. The transition period is still ongoing.
- For very complex animations or games, it is not the best tool.

**In one sentence:** the most pragmatic choice for most consumer applications.

### Flutter: the autonomous rendering engine

Flutter takes a radically different approach. Instead of using the phone's native components or a web browser, it **draws every pixel itself** through its own graphics engine, Impeller. Flutter brings its own paintbrush instead of borrowing the system's.

The result: a Flutter app looks exactly the same on an iPhone 16 and on an entry-level Android smartphone. Visual consistency is total.

In 2026, Impeller is the only rendering engine on iOS and the default engine on Android. Flutter compiles directly to machine code. It is not interpreted; it is compiled like a native application.

**Advantages:**

- The smoothest graphical rendering among cross-platform solutions: 90 to 120 frames per second.
- Identical interface across platforms: one design to maintain.
- Excellent developer experience, with instant reload while keeping app state.
- Compiled to machine code, with performance close to native.

**Limits:**

- The language, Dart, is less common than JavaScript. Hiring can be harder.
- The interface does not automatically "feel" like iOS or Android. You need to implement that explicitly.
- The package ecosystem is smaller than React Native's, although it is growing quickly.

**In one sentence:** the best choice when visual identity and smoothness are business priorities.

### Native (Swift/Kotlin): premium custom work

Native development means creating two separate applications: one in Swift for iOS and one in Kotlin for Android. Each uses the tools and interfaces provided directly by Apple and Google.

Both platforms have offered declarative frameworks for several years, SwiftUI since 2019 and Jetpack Compose since 2021, and these tools have now reached maturity. In 2026, they have become the de facto standard, making native development far more productive than older approaches.

**Advantages:**

- Immediate access to all new OS features, from the day they are released.
- Optimal performance: the fastest startup time and the lowest battery consumption.
- Deep ecosystem integration: Apple Watch, widgets, local AI, Siri, Google Assistant.
- The best option for apps that rely on on-device AI.

**Limits:**

- Two codebases to maintain: double effort, double budget.
- Senior Swift and Kotlin developers are rare and expensive.
- The time-to-market is the longest of the four approaches.

**In one sentence:** the gold standard when quality and long-term durability matter more than budget.

In 2026, a new criterion has entered the equation: **on-device artificial intelligence**. Apple Intelligence and Gemini Nano on Android mean smartphones now run AI models directly on their neural processors, without going through the cloud.

In practice, this enables:

- Real-time image recognition, without server latency.

If local AI is central to your product, for example in healthcare, real-time analysis or a voice assistant, native remains the only choice that guarantees optimal efficiency today. For secondary use cases, such as a chatbot or suggestions, all technologies can do the job.

## How should you choose? The decision tree

<ThemeImage
  lightSrc="/images/2026/appexpress/quelle-technologie-mobile-choisir/decision-tree.webp"
  darkSrc="/images/2026/appexpress/quelle-technologie-mobile-choisir/decision-tree.webp"

  width={1600}

Rather than comparing technical specs, ask yourself these questions in order:

**1. Does your team master web technologies (JavaScript, React, Vue)?**
→ Yes → **Capacitor**: build on existing skills without sacrificing mobile UX.
→ No, or not only → move to question 2.

**2. Is the app's visual identity a differentiating factor?**
→ Yes, design and animations are essential to our brand → **Flutter**.
→ No, functionality matters more → move to question 3.

**3. Do you need advanced phone features (local AI, sensors, Bluetooth LE)?**
→ Yes, they are core to the product → **Native**.
→ No, standard features are enough → move to question 4.

**4. Are budget and time-to-market your priorities?**
→ Yes → **React Native**: the best cost/quality/speed compromise.
→ No, I want the best possible result → **Flutter** or **Native**, depending on your need.

## Which profile are you?

**You are launching an MVP or an internal tool**

Move fast, validate the idea, iterate. **Ionic/Capacitor** lets you launch in a few weeks by using web skills. The mobile interface is built with dedicated components, not by copying a website. If the app succeeds, migrating to React Native or Flutter remains possible.

**You are building a consumer app**

You need a responsive interface, sustainable maintenance and a pool of developers to evolve the product. **React Native** is the most pragmatic choice. It is the most recruited technology in France.

**Your app IS your product**

The interface is a lever of differentiation. Your brand depends on smoothness, animations and custom design. **Flutter** gives you total control over every pixel, with an identical rendering on every device.

**You are building an advanced technology product**

On-device AI, augmented reality, deep integration with sensors or wearables. **Native** is the only path that will not impose any technical compromise.

## The trend to watch: Kotlin Multiplatform

A fifth player is emerging: **Kotlin Multiplatform (KMP)**. The principle is to share all business logic in Kotlin while keeping 100% native interfaces, SwiftUI on iOS and Compose on Android.

Companies such as Netflix and Cash App already use it in production. It is an attractive hybrid approach for ambitious projects that want the best of both worlds: the productivity of shared code and the interface quality of pure native.

It is still an early-adopter technology, but it may redefine the compromise in the coming years.

## What matters is the strategy

The classic trap is focusing on technology instead of product strategy. **The right choice is not the one that will be perfect forever, but the one that lets you ship now while keeping the ability to evolve your product.**

That is exactly the AppExpress approach: we do not sell a framework. We build a **delivery strategy** adapted to your context, budget and ambitions.

## Introduction: RAG as the backbone of applied AI

You have a product catalog with 50,000 references. An employee types a question into your internal chatbot: *"Which connector is compatible with the X400 series in a humid environment?"* A standard LLM, even GPT-4o or Claude, does not know your catalog. It will produce an answer that sounds plausible, but is wrong.

**RAG** (Retrieval-Augmented Generation) solves that problem. The principle is simple: before letting the LLM answer, the system **retrieves the relevant documents** from your data, then injects them into the model context. The LLM generates its answer from your real data, not from its training memory.

In 2026, RAG has become hard to ignore because the use cases have multiplied:

- **Customer support chatbots**: answers based on your documentation, FAQ and resolved tickets;
- **Internal document search**: a legal team searches for a precedent across 10,000 contracts;
- **HR assistant**: employees ask questions about leave policies, benefits and collective agreements;
- **Product knowledge base**: field technicians search for maintenance procedures;
- **Regulatory analysis**: GDPR compliance, ISO standards, legal texts.

The common point is clear: you own proprietary data, and you want an LLM to use it intelligently. You do not need to fine-tune a model every time a document changes. RAG works on live data.

But between Pinecone, pgvector, Qdrant, LangChain, Vertex AI and many other options, the choice has become confusing. This guide is here to make the trade-offs clearer.

## Before choosing a stack: the forgotten data audit

Before comparing vector databases or choosing a framework, there is a step that most RAG projects skip. It is also the first reason they fail.

As Jonas Roman, a RAG practitioner who has shipped more than 60 production projects, points out: **70 to 80% of the work in a RAG project is data preparation**, not prompts, models or architecture. It is the classic *garbage in, garbage out* problem. No vector database, no matter how powerful, can compensate for poorly prepared data.

Before writing code, audit your data across four dimensions:

1. **Quality**: are the documents readable? Are scanned PDFs usable? For complex documents such as diagrams or tables, advanced OCR tools like Mistral OCR provide better results than basic extractors.
2. **Structure**: is the data homogeneous? Three-line emails mixed with 200-page reports, inconsistent formats and contradictory information all need to be mapped upfront.
3. **Completeness**: do you actually have all the required data? A common issue is that business teams believe everything is available, but nobody has checked. Halfway through the project, a large portion turns out to be unusable.
4. **Accessibility**: where does the data live? Unstructured documents, SQL databases, ERP, CRM? This map determines the target architecture.

> **Practical advice**: a few hours of scoping with business experts can save weeks of development in the wrong direction. Invest that time before writing the first line of code.

## When RAG is not the answer

The reflex "internal data means vector RAG" has become automatic. But classic RAG, meaning embeddings plus semantic search, is not always the right tool. Before jumping into implementation, ask these questions.

**Does the answer already exist in structured data?**

If the question is *"What was revenue by region in 2023?"* and the data lives in an ERP or SQL database, you do not need RAG. A direct SQL query will be more reliable, faster and cheaper. RAG retrieves text passages; it does not aggregate, calculate or guarantee numerical accuracy.

**Do you need to read the whole corpus?**

*"What are the main lessons from our 30 audit reports for 2024?"* RAG is designed to find the most relevant passages, not to read an entire shelf. For this kind of global analysis, use **map-reduce** approaches, where each document is summarized first and the summaries are then synthesized, or use a multipass process.

**Do you need exhaustiveness?**

*"Give me every contract containing a non-compete clause."* Across 200 contracts, a vector RAG system might find 40 out of 47. The remaining 7 are missed. In legal or regulatory contexts, that gap is unacceptable. For these cases, you need **Knowledge Graphs** or reinforced hybrid search.

**Are calculations required?**

RAG does not calculate. If the answer requires arithmetic operations over data, create dedicated **tools**, such as calculation functions or API calls, that the LLM can trigger outside the RAG pipeline.

> **Good systems are almost always hybrid.** An upstream LLM classifier analyzes the user's intent and routes the request to the right tool: vector RAG for semantic understanding, SQL for structured data, tools for calculations, map-reduce for global synthesis. That router can simply be an LLM that classifies the question in a few seconds. You do not need complex infrastructure to start.

## The building blocks of RAG

A RAG system is not a single tool. It is a pipeline made of five distinct steps, each with its own technology choices.

### 1. Ingestion

Raw documents, whether PDFs, web pages, emails, Word files or SQL data, must be transformed into usable pieces. This is the **parsing** and **chunking** step.

- **Parsing**: extract text from a scanned PDF, interpret an HTML table, clean Markdown;
- **Chunking**: split a 200-page document into segments of 500 to 1,500 tokens, with or without overlap.

Chunking is more critical than it looks. Chunks that are too large dilute relevance. Chunks that are too small lose context. There is no universal size. It depends on your documents and your queries.

> **The 5 most expensive chunking mistakes**
>
> 1. **Cutting through a table**: a table split in two produces two unusable chunks. Detect tables and keep them together.
> 2. **Too much overlap**: 50% overlap doubles your index size without improving quality. Aim for 10 to 15%.
> 3. **Losing metadata**: chapter title, document name, date. These often disappear during chunking. Attach them systematically as chunk metadata.
> 4. **Ignoring document structure**: chunking a legal document by token count ignores articles and sections. Use the logical structure as delimiters.
> 5. **Using one format for every document**: a three-line email and a 200-page report should not be chunked the same way.

**Contextual embedding: enriching each chunk**

An emerging technique improves retrieval quality significantly: **contextual embedding**. Before vectorizing a chunk, ask an LLM to generate a short sentence that places the chunk in the broader document context. That sentence is added before embedding.

The result: each chunk "knows" where it comes from. A paragraph about the X400 pump specifications is no longer an orphaned fragment. It carries context such as "excerpt from the X400 pump maintenance manual, sealing chapter". The semantic surface area increases, and retrieval becomes much more precise.

### 2. Embedding

Each chunk is transformed into a **vector**, a list of numbers, typically 256 to 3,072 dimensions, encoding the semantic meaning of the text. Two sentences that mean the same thing will have close vectors, even if the words differ.

Common embedding models in 2026:

| Model | Dimensions | Vendor | Note |
|-------|------------|--------|------|
| `text-embedding-3-small` | 1,536 | OpenAI | Good price/quality ratio, widely used |
| `text-embedding-3-large` | 3,072 | OpenAI | More accurate, more expensive |
| `embed-v4` | 1,024 | Cohere | Strong multilingual performance |
| `voyage-3-large` | 1,024 | Voyage AI | Strong on code and technical documents |
| `voyage-multimodal-3` | 1,024 | Voyage AI | Text + image embeddings |
| `gemini-embedding-2` | 3,072 | Google | New in March 2026, natively multimodal |
| `mistral-embed` | 1,024 | Mistral AI | Very economical, solid general retrieval |
| `codestral-embed` | 3,072 | Mistral AI | Specialized for code |
| `nomic-embed-text-v2-moe` | 768 | Nomic AI | Open-source Mixture of Experts |
| `gte-qwen2-instruct` | 768-8192 | Alibaba | Open-source, variable dimensions, strong multilingual support |
| BGE-M3, E5-Mistral, GTE | Variable | Community | Free, self-hosted |

**Worth noting: Gemini Embedding 2**, released in March 2026, changes the landscape. It can place text, images, video, audio and PDFs in the **same vector space**. In practice, a search for "X400 connector wiring diagram" can retrieve both a documentation paragraph and a picture of the connector.

It leads MTEB benchmarks in English, multilingual and code tasks, supports more than 100 languages, and uses Matryoshka Learning to truncate vectors with limited quality loss. The downside: it is still in Public Preview and costs about 50% more than OpenAI for text-only use. For pure text, alternatives remain competitive. If your documents include images or diagrams, Gemini Embedding 2 has no direct competitor.

The embedding model has a direct impact on search quality. A multilingual model is better if documents mix English and French. A code-focused model performs better on technical documentation.

### 3. Vector storage

Vectors must be stored in a database capable of **similarity search**. This is the core architectural choice and the main section of this article.

### 4. Retrieval

When a user asks a question, the query is also transformed into a vector, then compared with stored vectors to find semantically close chunks.

Pure vector search has limits. Modern systems combine:

- **Semantic search**: vector similarity;
- **Full-text search**: BM25 and exact keywords, essential for names, product codes and reference numbers;
- **Hybrid search**: a weighted combination of both;
- **Reranking**: a specialized model such as Cohere Rerank or Jina Reranker reorders the results.

Hybrid search has become the production standard. Pure vector search misses queries with exact identifiers, such as *"invoice No.2024-0847"*. Full-text search misses queries phrased differently from the source document.

**How hybrid search works in practice**

The principle is simple: run two searches in parallel, vector and BM25 full-text, then merge the results. The most common fusion method is **Reciprocal Rank Fusion**:

1. Vector search returns one ranking: document A first, B third, C fifth.
2. BM25 returns another ranking: B first, D second, A fourth.
3. RRF combines them by weighting the inverse of the rank, with `k` usually around 60.

```text
score = 1 / (k + vector_rank) + 1 / (k + bm25_rank)
```

Document B, well ranked in both searches, moves to the top.

If a user searches *"X400 connector resistant to humidity"*, vector search captures meaning, for example "resistant to humidity" close to "waterproof" or "IP67", while BM25 catches the exact term "X400", which vector search might miss. Combining both retrieves the right document even when the query mixes natural language and technical terms.

### 4.5. Prompt engineering: the invisible block

Between retrieval and generation, many teams neglect prompt construction. Spending 80% of the effort on retrieval and then writing a basic prompt is a reliable way to get hallucinations, even with the right documents in context.

**Structure the prompt in separate blocks:**

1. **System**: the role of the AI, such as technical expert, legal assistant or customer support.
2. **Instructions**: strict rules, answer format, length, tone, and above all grounding: "Answer only from the provided documents. If the information is not in the context, say so explicitly."
3. **Context**: the chunks retrieved by the system, with metadata.
4. **Question**: the user's query.

**Three techniques that change the result:**

- **Mandatory citations**: require the LLM to cite sources for each claim. This reduces hallucinations and enables human verification.
- **Few-shot examples**: provide 2 or 3 complete examples of question, reasoning and expected answer.
- **Handling lost in the middle**: LLMs tend to ignore information in the middle of long contexts. Put the most relevant chunks at the beginning and the end.

### 5. Generation

The retrieved chunks are injected into the LLM prompt, and the model generates the final answer. The choice of LLM, GPT-4o, Claude, Gemini, Llama or Mistral, affects synthesis quality, but is usually the easiest part to swap.

This is where the most structural architectural decision happens. There are three broad families.

### Specialized databases

These tools are designed from the start for vector storage and search.

#### Pinecone

- **Deployment**: fully managed, serverless.
- **Strengths**: setup in minutes, no infrastructure to manage, automatic scaling, native hybrid search, namespaces for multi-tenancy.
- **Weaknesses**: proprietary, vendor lock-in, expensive at scale, no self-hosting, variable latency on serverless plans.
- **Pricing**: free up to 2 GB, then storage and reads are billed.
- **Best fit**: teams that want production RAG with no ops, scale-ups that value time-to-market.
- **Scale**: billions of vectors.

#### Qdrant

- **Deployment**: self-hosted or Qdrant Cloud.
- **Strengths**: written in Rust, fast, advanced metadata filtering, hybrid search, built-in quantization, simple API.
- **Weaknesses**: self-hosting requires DevOps, the cloud version is newer than Pinecone.
- **Pricing**: open-source self-hosted, cloud from about $25/month.
- **Best fit**: technical teams that want control without sacrificing performance, multi-cloud or on-premise.
- **Scale**: tested on billions of vectors, horizontal sharding.

#### Weaviate

- **Deployment**: self-hosted or Weaviate Cloud.
- **Strengths**: native GraphQL API, integrated vectorization, BM25 + vector hybrid search, generative search modules, good connector ecosystem.
- **Weaknesses**: heavier than Qdrant, GraphQL learning curve, integrated vectorization can create coupling.
- **Best fit**: projects that want a RAG pipeline integrated into the database.

#### Milvus / Zilliz

- **Deployment**: self-hosted with Milvus or managed with Zilliz Cloud.
- **Strengths**: built for massive scale, storage/compute separation, GPU acceleration, many index types, CNCF project.
- **Weaknesses**: complex to operate, depends on several components, overkill for small volumes.
- **Best fit**: very large volumes, DevOps-capable teams, multimodal search.

#### ChromaDB

- **Deployment**: embedded in Python or server mode.
- **Strengths**: easy installation, very approachable API, perfect for prototypes and notebooks.
- **Weaknesses**: not designed for large-scale production, no native hybrid search, no high availability.
- **Best fit**: prototypes, POCs, personal projects.

#### Turbopuffer

- **Deployment**: managed serverless.
- **Strengths**: stores vectors on object storage, very low idle cost, native BM25, good for variable workloads.
- **Weaknesses**: young product, potentially higher latency than in-memory databases, smaller ecosystem.
- **Best fit**: large corpora queried infrequently, semantic archive, constrained budgets.

### Extensions of existing databases

Already have a database? Add vector capabilities to it.

#### pgvector (PostgreSQL)

- **Deployment**: PostgreSQL extension available on Supabase, Neon, AWS Aurora, Railway or self-hosted PostgreSQL.
- **Strengths**: no new database to operate, vectors live next to relational data, standard SQL, joins with business tables, hybrid search via `tsvector` + pgvector.
- **Weaknesses**: slower than specialized databases beyond 10M vectors, HNSW indexing consumes RAM.
- **Best fit**: SMBs already on PostgreSQL, projects where vectors must be joined with business data.
- **Scale**: comfortable up to about 5-10M vectors.

**Example: semantic search with pgvector**

```sql
CREATE TABLE documents (
  id SERIAL PRIMARY KEY,
  title TEXT,
  content TEXT,
  embedding vector(1536)
);

CREATE INDEX ON documents
  USING hnsw (embedding vector_cosine_ops);

SELECT title, content,
       1 - (embedding <=> $1) AS similarity
FROM documents
ORDER BY embedding <=> $1
LIMIT 5;
```

#### MongoDB Atlas Vector Search

Transparent if you already run MongoDB Atlas. It combines vector search with filtering on MongoDB documents and a unified aggregation pipeline. The downside is that it is Atlas-only and less specialized than dedicated vector databases.

#### Redis Vector Search

Very fast because it runs in memory. Useful for semantic cache or low-latency RAG on moderate volumes. The cost grows quickly with RAM requirements.

#### Cloudflare Vectorize

Managed and integrated into Cloudflare Workers. Good for teams already building on the Cloudflare edge, but more limited than specialized vector databases.

#### Turso Vector

Distributed SQLite with vector support. Lightweight, edge-friendly and useful for embedded or mobile scenarios, but still limited for advanced vector workloads.

### Platform-integrated solutions

These solutions couple vector storage with a broader application platform.

#### Convex (@convex-dev/rag)

- **Deployment**: managed, integrated into Convex.
- **Strengths**: native vector support in a real-time database, no separate service, embeddings and search through Convex functions, real-time updates, hybrid search, end-to-end TypeScript.
- **Weaknesses**: tied to Convex, less control over indexing than Qdrant or Milvus.
- **Best fit**: real-time applications that need RAG without infrastructure complexity.

**Example: semantic search with Convex RAG**

```typescript
await rag.insertDocument(ctx, {
  title: "X400 pump maintenance procedure",
  body: "For the quarterly maintenance of the X400 pump...",
  metadata: { category: "maintenance", equipmentId: "X400" },
});

const results = await rag.search(ctx, {
  query: "pump maintenance humid environment",
  limit: 5,
  filter: { category: "maintenance" },
});
```

#### Firebase / Vertex AI RAG Engine

Managed on Google Cloud. Google handles ingestion, chunking, embedding, storage and retrieval, with native Gemini integration and Google Search grounding. It is powerful for companies already on Google Cloud, but comes with lock-in and complex pricing.

#### OpenAI File Search

Managed by OpenAI. Upload files and get a working RAG system in a few lines. The trade-off is that it is a black box: little control over chunking, embedding or retrieval, and costs can grow with large corpora.

## RAG frameworks: orchestration layer

Vector storage is necessary but not sufficient. In production, you need to orchestrate ingestion, chunking, retrieval, generation, errors, streaming and multi-turn flows.

### LangChain / LangGraph

The most popular RAG framework, available in Python and JavaScript. Its strengths are a huge ecosystem, connectors for almost everything, an active community and LangSmith for monitoring. Its weaknesses are abstraction overhead, debugging difficulty and frequent API changes. It is good for getting started, but many production teams end up using only a subset.

### LlamaIndex

Specialized in document indexing and retrieval. It is stronger than LangChain on complex ingestion, such as multi-column PDFs, tables and images, and its retriever abstractions are well designed. It is less general-purpose outside RAG.

### Haystack

A production-oriented RAG framework with declarative pipelines. Less fashionable, but explicit and reliable, with good support for evaluation.

### RAGatouille

A lightweight library centered on ColBERT. It can improve retrieval quality without changing the whole architecture, but it is not a full framework.

### Vercel AI SDK

A TypeScript/JavaScript SDK for building AI applications, with RAG support. It fits React, Next.js and streaming workflows well, and is a strong choice for TypeScript teams.

### Turnkey solutions

Vertex AI RAG Engine and OpenAI File Search manage the pipeline for you: upload, index, query, answer. Time-to-market is excellent, but control is low, lock-in is high, and cost can grow quickly.

## Google Vertex AI & Grounding

Google has taken a distinct approach that deserves its own section.

### Vertex AI Search

Grounding lets Gemini verify answers against the web in real time. When Gemini generates an answer, it can query Google Search to confirm facts and add citations.

This is useful when proprietary data must be enriched with current public information: news, regulation, market data.

### Vertex AI RAG Engine

A fully managed RAG pipeline:

1. Create a corpus and upload documents.
2. Google handles parsing, chunking, embedding and indexing.
3. Query through the API or directly from Gemini.

The promise is zero vector infrastructure. The reality is that it works well for standard cases, but lack of control over chunking and retrieval can be frustrating for complex business documents.

### When to choose Google

- You are already on Google Cloud.
- Gemini is your main LLM.
- Your documents are relatively standard.
- You need minimal time-to-market.

### When to build it yourself

- You need precise control over chunking.
- You want to remain LLM-agnostic.
- You have strict latency constraints.
- You need detailed control over hybrid search.

## Going further: often neglected aspects

### Evaluating RAG quality

Choosing a stack is one thing. Knowing whether it works is another. In production, you need to **measure** answer quality.

Automatic evaluation frameworks can quantify performance:

- **RAGAS**: measures faithfulness, relevance and completeness.
- **DeepEval**: adds metrics such as hallucination, toxicity and bias, with pytest integration.
- **Haystack Evaluation**: useful if you already use Haystack.

**The Golden Dataset**

Evaluation frameworks are useless without reference data. A **Golden Dataset** is a sample of real user questions, associated with expected answers and source documents. It is one of the most underrated and decisive tools in a RAG project.

Build it with business experts, not alone. Collect real questions across easy, medium and complex levels. Define the expected answer and output format for each. For a prototype, 10 to 50 questions are enough. For production, aim for 100 to 500.

Watch three metrics:

- **Precision**: is the answer exact, complete and reproducible?
- **Recall**: does retrieval find the right chunks?
- **Faithfulness**: does the answer stay grounded in the sources?
- **Latency**: users drop off above 3 to 5 seconds.

Change one variable at a time, then rerun the Golden Dataset. That is the only way to know what improves or degrades quality.

### Multimodal RAG

In 2026, companies no longer search only text. Technicians need **technical diagrams**, analysts need **financial charts**, training teams need **videos**.

Multimodal RAG is still young, but several approaches are emerging:

- **ColPali / ColQwen**: retrieval models that work directly on page images without OCR.
- **Milvus / Zilliz**: native support for multimodal vectors.
- **Multimodal LLMs**: GPT-4o, Gemini and Claude can analyze images in RAG context, but retrieving the right images remains the hard part.

If your business documents contain many visuals, multimodal is no longer a luxury. Expect a more complex pipeline and higher embedding costs.

### Privacy and sensitive data

For GDPR-regulated companies or teams handling sensitive information, ingestion must include **PII detection and anonymization** before embedding.

In practice:

- **Before embedding**: scan chunks for names, emails, phone numbers, IBANs and sensitive identifiers.
- **Anonymize or pseudonymize**: replace PII with generic tokens before creating vectors.
- **Tools**: Microsoft Presidio, AWS Comprehend, spaCy with custom NER models.

An embedding vector contains a semantic representation of the text. Sensitive information can theoretically leak from it. Upfront anonymization remains the most reliable protection.

### User interface

Most RAG guides stop at the technical pipeline. Yet the interface determines adoption.

**The classic chatbot is a trap.** An empty text box saying "Ask your question..." looks intuitive. In practice, users often do not know what to ask, how to phrase it, or what context to provide. The result is vague questions, imprecise answers and loss of trust.

**The alternative: guided business interfaces.** Instead of a text box, use dropdowns, structured fields and selection buttons. The interface captures 4 or 5 precise context points and sends structured JSON to the pipeline. The LLM then returns a structured answer: symptoms, causes, solutions, sources.

Why it works:

- **Lower cognitive load**: users do not have to guess the right phrasing.
- **Better retrieval**: a structured query with business context has a broader semantic surface.
- **UI-driven orchestration**: the interface can route to RAG, SQL or calculation based on user choices.
- **Adoption**: perceived RAG quality depends as much on the interface as on the pipeline.

> **Field feedback**: Jonas Roman observes that most high-performing production RAG projects do not use chatbots, but dedicated business interfaces that are faster to adopt and better suited to daily work.

### Observability

Evaluating RAG before launch is necessary. In production, you need **continuous monitoring**.

Place checkpoints at each critical step:

1. **After retrieval**: which chunks were returned? Are they relevant?
2. **After reranking**: did the order change? Are the right documents at the top?
3. **At the final answer**: are sources cited? Is the answer faithful to context?

When an answer is wrong, these checkpoints let you reverse-engineer the issue. Did retrieval return the wrong documents? Was the right document in context but ignored by the LLM? Was the prompt poorly structured?

**Reject black boxes for serious use cases.** RAG as a Service is convenient for prototypes, but you have little visibility into the internals. In finance, healthcare or legal, explainability is not a bonus. It is a requirement.

### Beyond vectors

RAG is not limited to "embeddings + vector database". Several emerging approaches complement or replace classic semantic search.

**GraphRAG / Knowledge Graphs**

Instead of chunking text and vectorizing chunks independently, GraphRAG preserves **relationships and logical links** between information. It is especially powerful when you need exhaustiveness: finding every contract with a clause, tracing process dependencies, identifying non-compliance across a corpus.

**Page Indexing**

A vectorless RAG technique that indexes documents by page and logical structure rather than by vectors. Useful when document layout carries as much meaning as content.

**Exploratory File Search agents**

Agents use multi-turn search to read through data like a human would. They ask an initial query, analyze results, refine the search and iterate until they find the information.

**Self-RAG / Reflective RAG**

The system retrieves information, then checks whether it is relevant and sufficient. If quality is too low, it launches new queries and only returns an answer after passing its own reliability criteria.

> **These approaches are still often R&D work.** Watch them closely, but be cautious before putting them into production. Classic vector RAG with hybrid search remains the proven standard.

## Recommendations by profile

### Early-stage startup

**-> Convex (@convex-dev/rag) or pgvector via Supabase**

You do not have DevOps capacity and want to ship quickly. Convex gives you a database, vector search and real-time behavior in one service. Supabase + pgvector is the alternative if you prefer PostgreSQL and your vector needs are simple.

Do not spend three months designing the perfect RAG system. Start simple and iterate.

### SMB with existing PostgreSQL

**-> pgvector**

You already run PostgreSQL in production, with a team that knows it and backups in place. Add pgvector as a SQL migration. Your vectors live next to your business data.

If you exceed 5 to 10 million vectors, or search latency becomes critical, it will be time to move to a specialized database.

### Scale-up / large volume

**-> Pinecone or Qdrant**

You have tens of millions of documents, sustained traffic and latency requirements. Pinecone if you want zero ops. Qdrant if you want control and have the team to operate it, or use Qdrant Cloud.

### Google Cloud enterprise

**-> Vertex AI RAG Engine + Qdrant/Pinecone for advanced cases**

Vertex AI handles standard cases. Qdrant or Pinecone handle cases that require fine control. Google Search grounding is a real differentiator when users need current public information.

### Multi-cloud enterprise

**-> Self-hosted Qdrant or Weaviate**

You do not want vendor lock-in. Qdrant and Weaviate run on any Kubernetes cluster. You keep full control over data and infrastructure.

### Fast prototype

**-> ChromaDB or OpenAI File Search**

You want to validate an idea in an afternoon. ChromaDB if you use Python and want to understand the mechanics. OpenAI File Search if you just want to see RAG working without managing vectors.

Do not put either one into large-scale production.

## Conclusion: RAG is not a single choice

RAG is not a product you buy. It is an **architectural spectrum**, from "three lines of code with OpenAI File Search" to "distributed pipeline with Qdrant, Cohere reranking and continuous evaluation".

The right choice depends on three factors:

1. **Where you are**: prototype, MVP or established production?
2. **What you have**: DevOps team, PostgreSQL already in place, Google Cloud?
3. **What you need**: volume, latency, accuracy?

**The worst mistake is over-architecting.** Complex multi-agent RAG architectures can look attractive, but field experience shows that many overly ambitious systems do not hold up in production. Each additional LLM call multiplies the error rate. If a model has a 5% error rate, cascading several calls noticeably lowers global reliability.

Start with the simplest solution that meets the immediate need. pgvector or Convex for a first production RAG. ChromaDB for a POC. Migrate when, and only when, you hit the limits.

The second worst mistake is underestimating upstream work. The best vector database in the world will not compensate for poorly prepared data, bad chunking or the wrong embedding model. Data preparation and business engineering represent most of the effort.

The third mistake is often invisible: neglecting the interface. A technically solid RAG hidden behind a plain text box will be underused. Think of RAG as a product for business users, not only as a technical pipeline.

**A word on trends**: keep an eye on object-storage architectures such as Turbopuffer, GraphRAG for cases requiring exhaustiveness and relationship traceability, and Self-RAG for more reliable agentic pipelines. RAG is not disappearing. It is evolving into a central knowledge infrastructure for enterprise AI.

Artificial intelligence is discussed constantly in companies. But in most companies, the reality is more down to earth: useful data, customers, projects, stock, invoices and field work, lives inside a business application that is only accessible from a web browser on a fixed workstation.

That is a good start. But it is only part of the equation.

A truly effective business application today relies on **three complementary layers**: the web interface for daily work, the mobile app for the field, and an artificial intelligence layer for automation.

## The web interface: the central workstation

This is the heart of the system. The web interface remains essential to:

- manage projects, contracts and invoices;
- track customer conversations and field work in real time;
- view dashboards and indicators;
- administer access rights and settings.

Whether it is a CRM, a job-site tracking tool, inventory management software or a team schedule, the principle is the same: the web interface is structured, complete and designed for office work. **The goal is not to replace it.** But it has limits: you need to be in front of a screen, connected, and know where to look for the information.

## The mobile app: field access

This is the layer that is most often missing. Yet it is the one that changes daily work the most for teams that are not behind a desk.

A craftsperson on site, a salesperson in a meeting, a technician on an intervention, a manager on the move: all of them need access to their data **where they are**, not only where their computer happens to be.

A well-designed mobile app makes it possible to:

- **receive push notifications**: new customer request, urgent ticket, overdue invoice, scheduled intervention;
- **validate quickly**: accept a quote, sign a document, confirm a job-site milestone;
- **check at a glance**: project status, customer history, stock level, daily schedule;
- **capture information in the field**: document photo, observation, voice note, quick comment;
- **stay responsive**: answer a customer in 2 minutes instead of 2 hours.

<Callout type="info" title="The key point">
  Mobile does not replace the web interface. It extends access to your business application beyond the office, where business actually happens.
</Callout>

It is not a "reduced version" of your software. It is the same database, the same access rights, the same business logic, but in a format suited to a pocket screen and mobile use.

## The AI layer: intelligent automation

Now imagine adding a third layer: structured access to your application's data, usable by an LLM, meaning a language model such as Claude or ChatGPT.

The principle is simple. Instead of navigating menus to find information, we create **dedicated access points** that AI can query. Not raw access to the whole database, but targeted, secure views that answer the most common business questions.

In practice, this can take the form of a command line tool or an MCP connector (Model Context Protocol), allowing a desktop AI assistant to query your business tool directly.

### What it enables

Some examples depending on the type of application:

**Customer management / CRM:**

- *"Summarize the Martin customer account: projects, payments, tickets"*
- *"Draft a follow-up email for each overdue invoice"*

**Job-site tracking:**

- *"Which job sites have fallen behind this week?"*
- *"List the interventions that have not been validated for more than 3 days"*

**Inventory management:**

- *"Which products are below the reorder threshold?"*
- *"Prepare a supplier order based on this month's sales"*

**Scheduling / HR:**

- *"Who is available next week for an intervention in Rennes?"*
- *"Summarize this month's overtime by team"*

<Callout type="tip" title="The real gain">
  The important thing is not response speed. It is that questions that were never asked, because they took too long to process, become possible.
</Callout>

## The three layers in action: a concrete scenario

Let us take a CRM as an example, although the principle applies to any business application.

**Monday morning, 8:00.** The AI automatically analyzes the application state and detects that an invoice for the Durand customer has been overdue for 7 days, that a support ticket has gone unanswered, and that the associated project has not moved forward for 10 days.

**8:05.** The manager receives a **push notification** on their phone: *"Durand customer: 3 attention points detected"*.

**8:06.** They open the **mobile app**, read the summary prepared by the AI, and approve in one tap the sending of a pre-written payment reminder email for the invoice.

**8:10.** Back at the office, the project manager opens the **web interface** to handle the ticket in depth, update the project schedule and document the exchange.

That is what the three layers do together.

## Tomorrow: autonomous agents

The three layers described above are already operational. But the natural evolution goes further: **autonomous AI agents**.

An agent is no longer an LLM waiting for a question. It is an assistant that **acts by itself** according to defined rules:

- it continuously monitors your business application;
- it detects situations that require action;
- it proposes or directly executes that action, with your validation.

For example: every morning, an agent can analyze overdue invoices, draft reminders, and submit them for approval through a mobile notification. Without being asked.

### For the more ambitious: OpenClaw

Platforms such as [OpenClaw](https://openclaw.ai) already make it possible to connect AI agents to your business tools: management application, email, calendar, messaging. The agent becomes a real digital teammate: it reads your data, reasons over it, and can act across several channels, email, Telegram, web interface.

This is no longer science fiction. It is an accessible reality, provided you have a well-structured database and secure access, exactly what the previous three layers put in place.

<Callout type="tip" title="The pragmatic approach">
  You do not start with an autonomous agent. You start with a solid application, add mobile, then add the AI layer. The agent arrives naturally when the foundations are in place.
</Callout>

## Why this approach works

Whether you are independent, a craft business, an SME or a larger organization, when resources are limited, **the same people often do everything**.

That is precisely why this approach works so well:

- **The web interface** structures work and centralizes data
- **The mobile app** keeps people responsive without tying them to the office
- **AI** handles repetitive tasks and surfaces what really matters

All of this with a **controlled cost**: we do not rebuild what already exists, we enrich it. Each layer is added without replacing the previous one.

## The right sequence to start

1. **A solid web application**: the foundation. Well-structured data, clear access rights, complete interface.
2. **A mobile application**: to extend access to the field and gain responsiveness.
3. **The AI layer**: start with 3 to 5 high-value use cases, test on real data, then expand.

<Callout type="warning" title="A point of vigilance">
  Security remains the priority. Each layer follows the same access rules: a salesperson only sees their customers, a contractor only sees their project. AI is not an exception.
</Callout>

# **BLE UScope – Advanced Bluetooth Low Energy Scanner and Analyzer**

BLE UScope is a Bluetooth Low Energy analysis tool for iOS, iPadOS, Android, and macOS Catalyst. The app scans nearby peripherals, reads advertisement data, tracks RSSI, then connects to explore their GATT structure.

Its goal is to make everyday BLE work more direct: inspect services and characteristics, read or write values, enable notifications, follow connection events, record monitoring sessions, and export usable traces.

## From scanning to GATT

The first screen is about observing the BLE environment in real time. BLE UScope shows discovered devices with RSSI, advertisement data, advertised services when available, and an estimated device category based on service UUIDs, manufacturer data, or the device name.

Once connected, the app moves into a more diagnostic-oriented detail view. You can browse services, characteristics, and descriptors, filter readable, writable, or notifiable characteristics, then inspect values as hexadecimal, UTF-8, or decimal data depending on the case. GATT properties stay visible, so you do not have to guess whether a characteristic supports `read`, `write`, `notify`, or `indicate`.

This part intentionally stays close to BLE terminology. When inspecting a peripheral, you need UUIDs, raw values, descriptors, and notification state to remain visible instead of being hidden behind a simplified UI.

## Monitoring, diagnostics, and exports

Scanning is only one part of the workflow. BLE UScope also lets you follow a connection over time: signal quality, connection duration, RSSI history, connection events, and received notifications.

BLE notifications can be recorded into monitoring sessions. This is useful when a characteristic emits a measurement, a state update, or a domain-specific event at a regular cadence. The app keeps values with their timestamp, device, service, characteristic, and RSSI when that information is available.

Diagnostics help validate a connected peripheral: service discovery, connection stability, characteristic reads, and simple response or throughput checks. Exports make it possible to keep a record of what was observed: device profile, GATT structure, characteristic values, logs, or notification sessions.

## Implementation

The Apple version is built with SwiftUI and CoreBluetooth. It supports iOS, iPadOS, and macOS through Catalyst, with layouts adapted to each screen size. The BLE layer handles scanning, multiple connections, CoreBluetooth restoration, reconnects, GATT exploration, and notification monitoring.

The Android version is built with Kotlin and Jetpack Compose. It uses `BluetoothGatt`, a sequential operation queue to avoid BLE operation collisions, state flows to synchronize the UI, Room for local data, and dedicated views for logs, favorites, history, and diagnostics.

Both versions share the same product intent while respecting each platform's constraints. On Android, that means dealing with modern Bluetooth permissions, scan throttling, and device-specific BLE stack behavior. On iOS, CoreBluetooth defines its own restoration, authorization, and lifecycle model.

## Data and privacy

BLE UScope runs locally. There is no account, no analytics, and no BLE data sent to a server. Favorites, custom names, histories, logs, and exports stay on the device unless you explicitly share or export them.

# Real-time sound levels with visual context

DecibelPeak starts from a simple idea: open the app, allow microphone access, and immediately see the sound level around you.

The current version keeps that direct reading, then adds more context: a circular gauge, waveform, spectrum, FFT bars, FFT circle, waterfall, dB curve, and dB peak view. It is not meant to replace a certified laboratory sound meter, but it is useful for observing an acoustic environment, comparing two situations, and keeping a real-time visual reference.

## What the app shows

The main view combines an instant dB value, a graduated gauge, and a sound-zone indicator. Levels are grouped into readable ranges, from quiet to loud, so the app does not leave you with a number alone.

The top area can cycle through several audio visualizations. The fast views show the waveform and spectrum, while the FFT and waterfall modes provide a more frequency-oriented look. The dB curve and dB peak modes are better suited for understanding changes over time: a gradual rise, a short event, the session average, or the highest value reached.

Local calibration lets you adjust the measurement offset in 0.5 dB steps. That helps bring DecibelPeak closer to a reference device or compensate for differences between phone microphones.
